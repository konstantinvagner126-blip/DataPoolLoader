# Module Store Draft

Черновик. Не является утвержденной частью roadmap/backlog.

## Ключевые решения

На текущий момент в черновике уже зафиксированы такие базовые решения:

- два режима работы:
  - `files`
  - `database`
- без гибридного режима;
- режимы независимы друг от друга и пересекаются только через общие абстракции;
- `database` режим работает только с PostgreSQL;
- при недоступной БД UI автоматически остается в `FILES`;
- в `database` режиме:
  - общая `current revision`
  - персональная `working copy`
  - `Save` сохраняет только personal draft
  - `Publish` создает новую общую ревизию;
- запуск в `database` режиме идет:
  - по `working copy`, если она есть и не `STALE`
  - иначе по `current revision`;
- для каждого запуска создается отдельный `execution snapshot`;
- поддерживается только однонаправленный поток:
  - `files -> database`
- импорт из файлов в БД выполняется через отдельный `ModuleSyncService`;
- история запусков в PostgreSQL проектируется только для режима `database`;
- cleanup истории запусков тоже относится только к режиму `database`;
- DB-модули можно:
  - создавать через UI
  - удалять из БД отдельной явной операцией.

## Идея

Ввести для UI два взаимоисключающих режима работы с модулями:

- `files`
- `database`

Гибридный режим не использовать.

## Зачем

- сохранить текущий файловый сценарий для тех, у кого нет доступа к БД;
- подготовить переход к хранению модулей в БД без немедленного отказа от `apps`;
- дать UI возможность работать с большим количеством модулей через централизованный реестр.

## Режим `files`

В этом режиме:

- UI работает только с каталогом `apps`;
- список модулей берется из файлов;
- редактирование идет по файлам;
- запуск идет по файлам;
- PostgreSQL для реестра модулей не требуется.

## Режим `database`

В этом режиме:

- UI работает только с PostgreSQL-реестром модулей;
- список модулей берется из БД;
- редактирование должно идти через БД;
- запуск должен идти из данных, хранящихся в БД;
- `appsRoot` нужен как источник для загрузки модулей в БД.

## Многопользовательская модель для режима `database`

Так как с PostgreSQL-реестром потенциально работают несколько пользователей, считаем правильным разделять:

- общее опубликованное состояние модуля;
- персональное черновое состояние каждого пользователя.

Из этого следуют правила:

- `module` и `current revision` являются общими для всех пользователей;
- `working copy` является персональной и принадлежит конкретному пользователю;
- обычный `Save` в режиме `database` не должен менять общее состояние модуля для всех;
- `Save` должен сохранять только персональный draft текущего пользователя;
- отдельная операция `Publish` должна создавать новую общую ревизию и переключать `current_revision_id`.

Это позволяет:

- не перетирать drafts друг друга;
- не блокировать параллельную работу нескольких пользователей;
- сохранить предсказуемую модель публикации изменений.

## Идентификация пользователя

На текущем этапе считаем правильным использовать учетную запись операционной системы.

Базовые поля идентичности:

- `actor_id` — логин ОС в том виде, в котором он получен из среды выполнения; основной идентификатор пользователя.
- `actor_source` — источник идентичности; на текущем этапе допустимы значения:
  - `OS_LOGIN`
  - `MANUAL_INPUT`
- `actor_display_name` — имя, которое показывается в UI; на первом этапе может совпадать с логином ОС.

Принцип:

- для DB-режима ключ пользователя строится по `actor_id`;
- `actor_source` храним отдельно на будущее, чтобы позже можно было перейти на SSO/LDAP без перелома схемы.

### Кросс-платформенное правило для `actor_id`

Так как хотим поддерживать работу не только на Linux, но и на macOS / Windows, фиксируем единый подход.

Предпочтительный источник идентичности:

- `actor_source = OS_LOGIN`

Способ получения `actor_id` должен быть кросс-платформенным и максимально простым.

Правильный порядок получения пользователя:

1. `System.getProperty(\"user.name\")` как основной кросс-платформенный источник.
2. fallback на переменные окружения:
   - `USER` для Linux/macOS;
   - `USERNAME` для Windows.
3. если пользователь автоматически не определился:
   - UI запрашивает `actor_id` у пользователя вручную;
   - для такого значения фиксируется `actor_source = MANUAL_INPUT`.

### Правило хранения `actor_id`

На текущем этапе нормализацию не делаем.

Правило:

- `actor_id` сохраняется в БД ровно в том виде, в котором он был получен из ОС;
- если `actor_id` введен вручную, он сохраняется в БД ровно в том виде, в котором его ввел пользователь;
- trim, lower-case, удаление доменной части и другие преобразования не выполняются.

Обоснование:

- это проще и прозрачнее;
- не появляется скрытая логика преобразования пользователя;
- для вашей модели важнее сохранить исходное значение, чем пытаться его унифицировать.

### Что хранить как `actor_display_name`

Для UI правильнее хранить отдельное отображаемое имя, но на первом этапе оно может совпадать с `actor_id`.

Предварительное правило:

- `actor_display_name` = raw OS login;
- если `actor_id` введен вручную, `actor_display_name` может быть равен введенному значению;
- на первом этапе `actor_display_name` и `actor_id` могут быть одинаковыми.

### Что делать, если пользователь не определился

Если ни один автоматический источник не дал валидного значения, режим `database` не должен молча падать в `FILES`.

Правильное поведение:

- UI должен показать форму ручного ввода `actor_id`;
- пока пользователь не ввел значение, режим `database` недоступен;
- после ручного ввода значение должно быть сохранено локально, чтобы не спрашивать его при каждом запуске;
- если пользователь отменил ввод или оставил значение пустым, UI остается в режиме `FILES`.

Предварительное правило хранения:

- вручную введенный `actor_id` хранится в UI storage текущей установки приложения;
- в PostgreSQL хранится уже итоговый `actor_id` и соответствующий `actor_source = MANUAL_INPUT`.

## Операции синхронизации

Для режима `database` должны быть предусмотрены явные операции:

- `Обновить все модули в БД`
- `Обновить выбранный модуль в БД`

Предварительное направление:

- единственный поддерживаемый источник импорта на текущем этапе — `appsRoot`;
- UI должен уметь переносить metadata, конфиг и SQL-ресурсы модуля в PostgreSQL;
- поддерживаем только поток `files -> database`;
- поток `database -> files` не рассматриваем;
- удаление `apps` откладывается на самый конец миграции.

Отдельно зафиксировано:

- синхронизация работает только с новыми и измененными модулями;
- удаленные из `apps` модули на этом этапе не отслеживаются;
- полная синхронизация не должна удалять и не должна архивировать записи в PostgreSQL.

## Переключение режима в UI

Предварительно зафиксировано:

- на странице модулей должен быть один переключатель режима работы;
- это не две обычные кнопки, а slider/switcher;
- состояния переключателя:
  - `FILES`
  - `DB`

Предварительное поведение:

- при запуске UI выполняется проверка подключения к PostgreSQL для режима `database`;
- если PostgreSQL доступен, переключатель позволяет выбрать `FILES` или `DB`;
- если PostgreSQL недоступен, состояние `DB` должно быть неактивным;
- пользователь должен видеть, что режим `DB` сейчас недоступен из-за отсутствия подключения.

Что считаем зафиксированным на уровне черновика:

- проверка доступности PostgreSQL нужна уже на старте приложения;
- доступность БД должна влиять на состояние переключателя;
- если БД недоступна, режим `DB` не должен быть доступен для выбора.

## Принцип полного разделения `files` и `database`

Предварительно считаем правильным не смешивать файловую и database-логику внутри одного режима работы.

Это правило считаем базовым:

- `files` и `database` — независимые режимы работы;
- каждый режим должен иметь собственную реализацию:
  - хранения модулей;
  - редактирования;
  - сохранения;
  - истории запусков;
  - служебных операций;
- пересечение функционала допустимо только там, где оно оформлено как обобщенный контракт, одинаково подходящий для обоих режимов.

Что это означает practically:

- не делаем общие таблицы или общий storage только ради формального единообразия;
- не переносим file-specific понятия в режим `database`;
- не переносим database-specific понятия в режим `files`;
- не пытаемся "сшить" режимы через большое количество условной логики.

Считаем правильным обобщать только такие сущности, как:

- `ModuleStore`
- `EditableModule`
- `ModuleSnapshot`
- `SqlAssetSnapshot`
- общий запуск через `core`

Все остальное должно оставаться независимым, пока для него не появится действительно общая и чистая абстракция.

Для этого нужен общий абстрактный слой хранения модулей:

- `ModuleStore`

Предварительные реализации:

- `FileModuleStore`
- `DatabaseModuleStore`

Что должно идти через этот слой:

- список модулей;
- загрузка деталей модуля;
- получение полного снимка модуля для запуска;
- сохранение изменений модуля;
- валидация содержимого модуля.

## Общая модель модуля

Чтобы запуск не зависел напрямую от файловой структуры, нужен общий снимок модуля.

Предварительное направление:

- `ModuleSnapshot`

В нем должны быть доступны как минимум:

- `id`
- metadata для UI;
- `configText`
- набор SQL-файлов и их содержимого;
- validation status;
- validation issues

Это должно позволить:

- в режиме `files` получать snapshot из `apps`;
- в режиме `database` получать snapshot из PostgreSQL;
- запускать модуль из одинаковой модели, независимо от режима.

## Контракт `ModuleStore`

Предварительно считаем правильным, что UI не должен напрямую знать, работает он с файлами или с PostgreSQL.

Для этого нужен единый контракт:

- `ModuleStore`

### Что должен уметь `ModuleStore`

Минимальный набор операций:

- `listModules`
  - вернуть список модулей для каталога UI
- `getEditableModule`
  - вернуть модуль в виде, пригодном для редактора
- `saveEditableModule`
  - сохранить изменения модуля в выбранное хранилище
- `publishWorkingCopy`
  - опубликовать персональную рабочую копию в новую общую ревизию
- `getModuleSnapshot`
  - вернуть неизменяемый снимок модуля для запуска

Отдельно фиксируем:

- операции синхронизации `files -> database` не входят в `ModuleStore`;
- для них остается отдельный `ModuleSyncService`.
- в режиме `database` операции store должны быть actor-aware:
  - store должен понимать, чей именно draft открывается, сохраняется, публикуется и запускается.
  - `getEditableModule` должен возвращать:
    - персональную `working copy` пользователя, если она существует;
    - иначе `current revision`.
  - `getModuleSnapshot` должен собираться:
    - из персональной `working copy` пользователя, если она существует и не `STALE`;
    - иначе из `current revision`.

## Модели, которые должен отдавать `ModuleStore`

Чтобы не смешивать задачи списка, редактора и запуска, считаем правильным иметь три разные модели.

### 1. `ModuleCatalogItem`

Назначение:

- минимальная карточка модуля для списка в UI.

Что должно быть в модели:

- `moduleId` — стабильный идентификатор модуля.
- `title` — отображаемое имя модуля.
- `description` — краткое описание модуля.
- `tags` — список тегов для поиска и фильтрации.
- `hiddenFromUi` — нужно ли скрывать модуль из общего списка.
- `validationStatus` — статус валидности.
- `validationIssues` — краткий список проблем и предупреждений.

### 2. `EditableModule`

Назначение:

- состояние модуля для экрана редактирования в UI.

Что должно быть в модели:

- `moduleId` — идентификатор модуля.
- `title` — отображаемое имя.
- `description` — описание.
- `tags` — теги.
- `hiddenFromUi` — признак скрытия в UI.
- `validationStatus` — статус валидности.
- `validationIssues` — полный список проблем и предупреждений.
- `configText` — текст `application.yml` в редакторе.
- `sqlAssets` — список SQL-артефактов для вкладки SQL.

Отдельно важно:

- `EditableModule` нужен для UI-редактора;
- store сам решает, как получить этот вид:
  - `FileModuleStore` — из файлов;
  - `DatabaseModuleStore` — из БД и сохраненного канонического YAML-представления.

### 3. `ModuleSnapshot`

Назначение:

- неизменяемый снимок модуля для запуска.

Что должно быть в модели:

- `moduleId` — идентификатор модуля.
- `versionToken` — стабильный идентификатор текущего снимка.
  - в режиме `database` это может быть `revisionId`;
  - в режиме `files` это может быть hash или synthetic token.
- `title` — имя модуля.
- `validationStatus` — статус валидности.
- `validationIssues` — список проблем.
- `configText` — итоговый текст `application.yml`, который будет использоваться для запуска.
- `sqlAssets` — итоговый набор SQL-артефактов с содержимым.

Почему это важно:

- запуск должен зависеть только от `ModuleSnapshot`;
- раннер не должен знать, получен ли snapshot из файлов или из PostgreSQL.

### 4. `SqlAssetSnapshot`

Назначение:

- единая модель SQL-артефакта для редактора и запуска.

Что должно быть в модели:

- `assetKey` — стабильный ключ артефакта внутри модуля.
- `label` — отображаемое имя, например `Общий SQL` или `Источник: db1`.
- `sqlText` — текст SQL.
- `originKind` — происхождение SQL: `INLINE` или `FILE`.
- `originPath` — исходный путь файла, если он был.

## Что считаем важным в контракте

- `ModuleStore` не должен отдавать наружу file-path-специфику как обязательную часть API;
- `ModuleStore` не должен заставлять UI знать про таблицы PostgreSQL;
- `ModuleSnapshot` должен быть достаточен для запуска без дополнительного чтения из файлов или БД;
- `EditableModule` должен быть достаточен для полного UI-редактирования.

## Жизненный цикл ревизий

Предварительно считаем правильным, что в режиме `database` должны существовать два уровня состояния:

- рабочее состояние модуля (`working copy`);
- зафиксированная ревизия (`current revision`).

Ревизии при этом остаются неизменяемыми.

Это означает:

- существующая ревизия после создания не редактируется;
- обычное сохранение в UI обновляет персональное рабочее состояние пользователя;
- модуль указывает на одну текущую активную ревизию через `current_revision_id`.

## Что такое `current_revision`

`current_revision` — это ревизия, которую UI считает основной для модуля в данный момент.

Именно она:

- является общей опубликованной версией модуля для всех пользователей;
- является базовым состоянием модуля, если у текущего пользователя нет `working copy`;
- используется для запуска модуля, если у текущего пользователя нет `working copy`;
- показывается в списке модулей как актуальная версия.

## Что такое `working copy`

`working copy` — это текущее редактируемое состояние модуля в режиме `database`.

Именно оно:

- принадлежит конкретному пользователю;
- меняется при сохранении из UI этим пользователем;
- может обновляться много раз подряд без создания новых ревизий;
- служит рабочей версией, с которой пользователь взаимодействует в редакторе.

## Когда создается новая ревизия

Новая ревизия должна создаваться в следующих случаях:

- синхронизация модуля из `apps` в PostgreSQL выявила новое или измененное содержимое;
- пользователь явно публикует свою `working copy`.

То есть:

- `sync from files` создает новую ревизию;
- `Publish` создает новую ревизию.
- обычный `UI save` новую ревизию не создает.

## Что происходит при сохранении

Предварительное поведение:

1. UI редактирует `EditableModule`.
2. Backend валидирует содержимое.
3. Backend обновляет персональную `working copy` текущего пользователя в режиме `database`.

Итог:

- пользователь видит сохраненные изменения;
- другие пользователи не получают эти изменения автоматически;
- новые ревизии не плодятся при каждом обычном сохранении.

## Что происходит при публикации

Предварительное поведение:

1. Пользователь явно выполняет `Publish`.
2. Backend проверяет, что `working copy.base_revision_id == module.current_revision_id`.
3. Если проверка проходит:
   - создается новая `module_revision`;
   - `module.current_revision_id` переключается на нее;
   - рабочая копия публикующего пользователя может быть удалена как уже примененная.
4. Если проверка не проходит:
   - публикация отклоняется;
   - `working copy` помечается как `STALE`;
   - пользователь должен сначала разрешить конфликт с новой опубликованной ревизией.

Это означает:

- публикация работает по optimistic concurrency;
- shared state меняется только через явный publish;
- обычный save shared state не меняет.

## Что происходит при запуске

Предварительное поведение:

- если у текущего пользователя есть его `working copy` со статусом `CLEAN` или `DIRTY`, запуск идет по ней;
- если у текущего пользователя нет `working copy`, запуск идет по `current_revision_id`;
- если у текущего пользователя `working copy` имеет статус `STALE`, UI не должен запускать ее молча:
  - пользователь должен явно подтвердить запуск устаревшей рабочей копии;
  - либо отбросить `working copy`;
  - либо пересоздать ее от новой `current revision`;
- перед запуском должен создаваться отдельный execution snapshot;
- раннер получает `ModuleSnapshot`, собранный из execution snapshot, а не из "живого" редактируемого состояния;
- это делает запуск воспроизводимым: всегда можно понять, какая именно версия модуля была реально запущена.

## Почему ревизии должны быть immutable

Причины:

- можно разбирать, какой именно конфиг участвовал в конкретном запуске;
- можно откатываться на предыдущую ревизию;
- sync из файлов не уничтожает прошлое состояние;
- UI и логика запуска становятся предсказуемее.

## Что считаем правильным на текущем этапе

- делать ревизии immutable;
- не обновлять старую ревизию inplace;
- не создавать новую ревизию на каждый `UI save`;
- использовать персональную `working copy` как основное редактируемое состояние;
- менять `current_revision_id` только при появлении новой зафиксированной ревизии.
- shared state менять только через явную публикацию;
- если у текущего пользователя `working copy` существует и не имеет статуса `STALE`, запускать модуль по ней;
- если у текущего пользователя `working copy` имеет статус `STALE`, требовать явного решения пользователя;
- если у текущего пользователя `working copy` отсутствует, запускать по `current revision`;
- для каждого запуска сохранять отдельный execution snapshot.

## Lifecycle DB-модуля

### Создание нового модуля через UI

Рекомендация:

- создавать новый модуль через UI только в режиме `database`;
- сценарий создания делать отдельным экраном, а не встроенным блоком на странице редактирования.

Обоснование:

- создание модуля — отдельная служебная операция;
- основной экран модулей не должен перегружаться полями первичной инициализации;
- в multi-user модели важно сразу инициализировать baseline и personal draft правильно.

### Где размещать создание модуля

На странице модулей в режиме `database` должна быть отдельная кнопка:

- `Создать модуль`

Эта кнопка ведет на отдельную страницу:

- `Новый модуль`

На странице `Новый модуль` должны быть:

- заголовок `Новый модуль`
- кнопка `К модулям`
- форма первичного создания
- кнопка `Создать модуль`
- кнопка `Отмена`

### Какие поля обязательны при создании

На первом этапе считаем достаточным такой набор.

Обязательные поля:

- `moduleCode`
- `title`

Необязательные поля:

- `description`
- `tags`
- `hiddenFromUi`

Правила по умолчанию:

- `hiddenFromUi = true`

Это безопасный default:

- новый DB-модуль не появляется сразу в обычном пользовательском списке;
- создатель может спокойно донастроить его в своей `working copy`;
- когда модуль готов, он сам решает, нужно ли показывать его в UI.

### Проверки при создании модуля

Backend должен валидировать:

- что режим `database` активен;
- что PostgreSQL доступна;
- что `moduleCode` не пустой;
- что `title` не пустой;
- что `moduleCode` уникален в DB-реестре.

Отдельно фиксируем:

- наличие или отсутствие файлового модуля с тем же `moduleCode` при создании DB-модуля не учитывается;
- на этапе создания считаем, что DB-модуль создается в собственной предметной области, без оглядки на `appsRoot`.

### Какой baseline создается при создании

При успешном создании backend должен сразу создавать:

- `module`
- первую `module_revision`
- персональную `working copy` создателя как копию первой ревизии

При этом для нового модуля сразу фиксируется:

- `module_origin_kind = CREATED_IN_UI`

То есть модуль появляется сразу в полноценном lifecycle, а не как "пустая оболочка без состояния".

### Какой шаблон использовать для первой ревизии

На первом этапе нужен один шаблон:

- `Пустой модуль`

Состав baseline:

- metadata из формы создания;
- `hiddenFromUi = true` по умолчанию;
- `sources = []`
- `quotas = []`
- `target.enabled = false`
- без SQL-assets
- остальные scalar-поля конфигурации заполняются безопасными дефолтами runtime-модели.

Это означает:

- первая ревизия может быть `INVALID`;
- это нормально;
- модуль считается созданным, но еще не готовым к запуску, пока пользователь не заполнит конфиг.

### Какие дефолты использовать в первой ревизии

На первом этапе считаем правильным использовать дефолты текущей runtime-модели приложения.

Принцип:

- не изобретать второй набор defaults только для DB-режима;
- initial revision должна собираться из тех же значений по умолчанию, что и обычный новый конфиг в UI.

Отдельно фиксируем:

- output-path и другие derived значения могут рассчитываться по существующим правилам UI;
- если часть значений определяется только после редактирования, это допустимо.

### Что происходит после создания

После успешного создания UI должен:

1. Показать success-сообщение.
2. Перенаправить пользователя на страницу редактирования только что созданного модуля.
3. Открыть именно его personal `working copy`.
4. Показать статус:
   - модуль создан;
   - опубликован baseline;
   - у пользователя есть personal draft.

### Что считать неуспешным созданием

Создание модуля должно целиком отклоняться, если:

- `moduleCode` уже существует в БД;
- не удалось создать первую ревизию;
- не удалось создать personal `working copy`.

Никаких частично созданных модулей оставлять нельзя.

Операция должна быть атомарной.

### Что считаем неправильным

Не хотим:

- создавать DB-модуль прямо из карточки существующего модуля;
- создавать модуль без baseline revision;
- создавать модуль без personal `working copy` создателя;
- автоматически делать новый модуль видимым всем пользователям по умолчанию.

### Удаление модуля из БД

Рекомендация:

- возможность удаления модуля нужна только в режиме `database`;
- удаление должно работать как явная пользовательская операция;
- удаление должно выполняться только для одного модуля за раз.

Обоснование:

- пользователь должен иметь возможность удалить ненужный DB-модуль;
- особенно это важно для модулей, созданных вручную через UI;
- операция удаления не должна быть частью `sync`, это отдельный lifecycle action.

### Где размещать удаление

На странице редактирования DB-модуля должна быть отдельная кнопка:

- `Удалить модуль из БД`

Эта кнопка:

- показывается только в режиме `database`;
- не показывается в режиме `files`;
- должна требовать явного подтверждения.

### Как должно работать удаление

Считаем правильным делать hard delete из DB-реестра.

При удалении модуля должны удаляться:

- `module`
- все `module_revision`
- все `module_working_copy`
- все `execution_snapshot`
- вся история запусков модуля:
  - `module_run`
  - `module_run_source_result`
  - `module_run_event`
  - `module_run_artifact`

Это должно происходить транзакционно.

### Когда удаление должно быть запрещено

Удаление модуля не должно выполняться, если:

- по модулю сейчас идет `syncOneFromFiles`;
- идет `syncAllFromFiles`;
- по модулю сейчас выполняется активный запуск в режиме `database`.

В этих случаях UI должен показывать понятную причину отказа.

### Подтверждение удаления

Перед удалением UI должен показывать confirm-диалог.

Минимально в нем должно быть:

- `moduleCode`
- `title`
- происхождение модуля:
  - `IMPORTED_FROM_FILES`
  - `CREATED_IN_UI`
- предупреждение о том, что история запусков и ревизии будут удалены.

### Особое правило для импортированных модулей

Если `module_origin_kind = IMPORTED_FROM_FILES`, UI должен показывать дополнительное предупреждение:

- `Этот модуль был импортирован из файлов. Если файловый модуль по-прежнему существует в appsRoot, следующий import может создать его заново.`

Это не должно блокировать удаление, но пользователь должен понимать последствие.

### Что считать успешным удалением

Удаление модуля считаем успешным, если:

- запись `module` удалена;
- каскадно удалены связанные ревизии, drafts, execution snapshots и run-history;
- UI после ответа backend удаляет модуль из DB-каталога на экране.

### Что считать неправильным

Не хотим:

- soft delete вместо реального удаления;
- массовое удаление нескольких DB-модулей на первом этапе;
- молча удалять модуль без явного подтверждения;
- блокировать удаление только из-за того, что модуль когда-то был импортирован из файлов.

### Отдельная сущность `working copy`

Рекомендация:

- `working copy` не моделировать как ревизию;
- хранить ее отдельно как mutable draft.

Предварительные поля:

- `working_copy_id`
- `module_id`
- `owner_actor_id`
- `owner_actor_source`
- `owner_actor_display_name`
- `base_revision_id`
- `working_copy_json`
- `working_copy_yaml`
- `status`
- `content_hash`
- `updated_at`

Где `status`:

- `CLEAN`
- `DIRTY`
- `STALE`

Обоснование:

- ревизия — immutable baseline;
- working copy — mutable draft;
- смешивать эти две природы в одной сущности неправильно.

Дополнительные правила жизненного цикла:

- `working copy` создается лениво:
  - при первом сохранении из UI в режиме `database`;
  - при создании нового модуля через UI;
  - при rollback, если пользователь восстанавливает старую ревизию в рабочее состояние.
- обычный sync из файлов `working copy` не создает.
- если `working copy` после сохранения полностью совпадает с `current revision`, ее допустимо удалить как избыточную.
- у одного пользователя может быть только одна активная `working copy` на один модуль.
- статус `working copy` должен интерпретироваться так:
  - `CLEAN` — рабочая копия совпадает с базовой ревизией, и базовая ревизия совпадает с `current revision`;
  - `DIRTY` — рабочая копия отличается от базовой ревизии, но базовая ревизия все еще равна `current revision`;
  - `STALE` — базовая ревизия рабочей копии больше не равна `current revision`.

## Конкретная доменная модель для PostgreSQL

Под существующий вид `application.yml` считаем правильным не хранить модуль в одной blob/yaml-колонке как источник истины.

Каноническая модель должна быть структурированной.

### 1. Сущность `module`

Назначение:

- стабильная идентичность модуля;
- указатель на текущую активную ревизию.
- хранение происхождения модуля на уровне реестра.

Описание полей:

- `module_id` — внутренний уникальный идентификатор модуля в БД.
- `module_code` — стабильный код модуля, по которому UI и пользователь его различают; аналог имени каталога модуля в `apps`.
- `module_origin_kind` — как модуль впервые появился в БД:
  - импортирован из файлов;
  - создан вручную через UI.
- `current_revision_id` — ссылка на текущую активную ревизию модуля, которую UI считает основной.
- `created_at` — дата и время создания записи модуля в реестре.
- `updated_at` — дата и время последнего изменения записи модуля или переключения его текущей ревизии.

Важно:

- `module_origin_kind` описывает первое появление модуля в БД;
- это поле не должно переписываться при последующих sync или publish;
- источник конкретной ревизии по-прежнему хранится отдельно в `revision_source`.

### 2. Сущность `module_revision`

Назначение:

- неизменяемая ревизия модуля;
- именно ревизия является unit запуска.

В ревизии должны храниться scalar-поля текущего `app`-конфига:

Описание полей:

- `revision_id` — внутренний уникальный идентификатор ревизии.
- `module_id` — ссылка на модуль, к которому относится ревизия.
- `revision_no` — порядковый номер ревизии внутри одного модуля.
- `created_at` — дата и время создания ревизии.
- `created_by` — кто создал ревизию: пользователь, sync-процесс или системная операция.
- `revision_source` — источник появления ревизии: синхронизация из файлов, публикация рабочей копии или создание нового модуля.

UI metadata:

- `title` — отображаемое имя модуля в UI.
- `description` — пользовательское описание модуля в UI.
- `tags` — список тегов для фильтрации и поиска в UI.
- `hidden_from_ui` — признак, нужно ли скрывать модуль из обычного пользовательского списка.

Validation:

- `validation_status` — итоговый статус валидации ревизии: валидна, с предупреждениями или невалидна.
- `validation_issues` — список найденных проблем и предупреждений в сериализованном виде.

Поля `app` верхнего уровня:

- `output_dir` — каталог, куда модуль пишет выходные файлы.
- `file_format` — формат выходных файлов; сейчас фактически ожидается `csv`.
- `merge_mode` — режим объединения данных: `PLAIN`, `ROUND_ROBIN`, `PROPORTIONAL`, `QUOTA`.
- `error_mode` — стратегия обработки ошибок; в текущей модели фактически поддерживается `CONTINUE_ON_ERROR`.
- `parallelism` — максимальное количество параллельных выгрузок по источникам.
- `fetch_size` — JDBC fetch size для потокового чтения данных из БД.
- `query_timeout_sec` — таймаут SQL-запроса в секундах, если задан.
- `progress_log_every_rows` — через сколько строк логировать прогресс выгрузки.
- `max_merged_rows` — ограничение на размер итогового merged-файла, если задано.
- `delete_output_files_after_completion` — нужно ли удалять выходные файлы после завершения сценария.

Дополнительно:

- `snapshot_json` — полный снимок конфигурации ревизии в JSON для стабильного runtime-восстановления.
- `snapshot_yaml` — полный снимок конфигурации ревизии в YAML для экспорта, совместимости и отладки.
- `content_hash` — контрольная сумма содержимого ревизии для определения изменений при sync.

### 3. Сущность `module_revision_source`

Назначение:

- список `sources` конкретной ревизии.

Поля:

Описание полей:

- `revision_id` — ссылка на ревизию, к которой относится источник.
- `source_name` — логическое имя источника из конфига (`db1`, `db2` и т.д.).
- `sort_order` — порядок отображения и сериализации источников.
- `jdbc_url_expr` — JDBC URL источника в исходном виде, без разрешения placeholder-ов.
- `username_expr` — логин источника в исходном виде; ожидается placeholder/expression.
- `password_expr` — пароль источника в исходном виде; ожидается placeholder/expression.
- `sql_asset_id` — ссылка на SQL-артефакт, который нужно выполнять для этого источника.

Важно:

- храним не resolved values, а именно raw expression из конфига;
- placeholders вида `${...}` должны сохраняться как есть.
- реальные логины и пароли в PostgreSQL не храним;
- для `username_expr` и `password_expr` в БД должны храниться только ссылки вида `${...}`;
- значения логина и пароля для запуска должны браться из `credential.properties`.

### 4. Сущность `module_revision_target`

Назначение:

- `target` конкретной ревизии.

Поля:

Описание полей:

- `revision_id` — ссылка на ревизию, к которой относится target.
- `enabled` — включен ли импорт в target для данной ревизии.
- `jdbc_url_expr` — JDBC URL target в исходном виде, без разрешения placeholder-ов.
- `username_expr` — логин target в исходном виде; ожидается placeholder/expression.
- `password_expr` — пароль target в исходном виде; ожидается placeholder/expression.
- `table_name` — имя целевой таблицы для загрузки данных.
- `truncate_before_load` — нужно ли выполнять `TRUNCATE TABLE` перед загрузкой.

Важно:

- для target также не храним реальные логины и пароли;
- в БД сохраняются только выражения/placeholder-ы;
- разрешение credentials при запуске идет через `credential.properties`.

### 5. Сущность `module_revision_quota`

Назначение:

- `quotas` конкретной ревизии.

Поля:

Описание полей:

- `revision_id` — ссылка на ревизию, к которой относится квота.
- `source_name` — имя источника, для которого задана квота.
- `percent` — процент участия источника в режиме `QUOTA`.
- `sort_order` — порядок хранения и отображения квот.

### 6. Сущность `module_revision_sql_asset`

Назначение:

- хранение SQL в БД как отдельных артефактов.

Поля:

Описание полей:

- `sql_asset_id` — внутренний уникальный идентификатор SQL-артефакта.
- `revision_id` — ссылка на ревизию, к которой относится SQL-артефакт.
- `asset_kind` — тип SQL-артефакта: общий SQL или SQL конкретного источника.
- `asset_key` — ключ артефакта внутри ревизии; например `common` или имя источника.
- `label` — человекочитаемая подпись для UI, например `Общий SQL` или `Источник: db1`.
- `sql_text` — фактический текст SQL, который будет использоваться в runtime.
- `origin_kind` — происхождение SQL: был ли он задан inline или загружен из файла.
- `origin_path` — исходный путь SQL-файла, если SQL пришел из `commonSqlFile` или `sqlFile`.
- `sort_order` — порядок хранения и отображения SQL-артефактов.
- `content_hash` — контрольная сумма SQL-текста для определения изменений.

### 7. Сущность `module_sync_run`

Назначение:

- аудит и диагностика синхронизаций `files -> database`.

Описание полей:

- `sync_run_id` — внутренний уникальный идентификатор операции синхронизации.
- `started_at` — дата и время старта синхронизации.
- `finished_at` — дата и время завершения синхронизации.
- `started_by_actor_id` — идентификатор пользователя, который инициировал sync, если sync запущен вручную.
- `started_by_actor_source` — источник идентичности пользователя, инициировавшего sync.
- `started_by_actor_display_name` — отображаемое имя инициатора sync.
- `status` — итоговый статус синхронизации: успешно, с ошибкой, частично успешно.
- `scope` — область синхронизации: все модули или один выбранный модуль.
- `module_code` — код модуля, если запускалась точечная синхронизация одного модуля.
- `details` — технические детали результата синхронизации в сериализованном виде.

### 8. Сущность `module_working_copy`

Назначение:

- текущее редактируемое состояние модуля в режиме `database`;
- mutable draft, который не является ревизией.

Описание полей:

- `working_copy_id` — внутренний уникальный идентификатор рабочей копии.
- `module_id` — ссылка на модуль, к которому относится рабочая копия.
- `owner_actor_id` — идентификатор пользователя, которому принадлежит рабочая копия.
- `owner_actor_source` — источник идентичности пользователя; на текущем этапе:
  - `OS_LOGIN`
  - `MANUAL_INPUT`
- `owner_actor_display_name` — отображаемое имя владельца рабочей копии.
- `base_revision_id` — ревизия, от которой была создана рабочая копия.
- `status` — состояние рабочей копии: чистая, измененная или устаревшая.
- `working_copy_json` — полный JSON-снимок рабочей копии для runtime и формы UI.
- `working_copy_yaml` — канонический YAML рабочей копии для текстового редактора.
- `content_hash` — контрольная сумма рабочей копии.
- `created_at` — дата и время создания рабочей копии.
- `updated_at` — дата и время последнего изменения рабочей копии.

Важно:

- `module_working_copy` не раскладывается на дочерние таблицы `sources/quotas/target`;
- это намеренный компромисс: mutable draft храним денормализованно;
- структурированной и аналитически удобной остается модель ревизий, а не черновиков.

### 9. Сущность `execution_snapshot`

Назначение:

- точное состояние конфигурации, которое реально было передано в запуск.

Описание полей:

- `execution_snapshot_id` — внутренний уникальный идентификатор снапшота запуска.
- `module_id` — ссылка на модуль, для которого создан snapshot.
- `requested_by_actor_id` — идентификатор пользователя, который инициировал запуск.
- `requested_by_actor_source` — источник идентичности пользователя, инициировавшего запуск.
- `requested_by_actor_display_name` — отображаемое имя пользователя, инициировавшего запуск.
- `source_revision_id` — ревизия, на основе которой создан snapshot, если запуск шел по ревизии.
- `source_working_copy_id` — рабочая копия, на основе которой создан snapshot, если запуск шел по ней.
- `snapshot_json` — полный JSON-снимок запускаемой конфигурации.
- `snapshot_yaml` — полный YAML-снимок запускаемой конфигурации.
- `content_hash` — контрольная сумма снапшота запуска.
- `created_at` — дата и время создания execution snapshot.

Важно:

- execution snapshot отделен от ревизий и рабочей копии;
- запуск должен ссылаться на snapshot, а не на изменяемое текущее состояние.

### 10. Сущность `module_sync_run_item`

Назначение:

- детализированный результат синхронизации по каждому модулю внутри одного `module_sync_run`.

Описание полей:

- `sync_run_item_id` — внутренний уникальный идентификатор записи.
- `sync_run_id` — ссылка на общую операцию синхронизации.
- `module_code` — код модуля, который обрабатывался в рамках sync.
- `action` — что именно произошло с модулем: создан, обновлен, пропущен, завершился ошибкой.
- `status` — итоговый статус обработки этого модуля.
- `detected_hash` — hash содержимого, вычисленный по файлам на момент sync.
- `result_revision_id` — ссылка на созданную ревизию, если sync привел к появлению новой ревизии.
- `details` — технические детали по конкретному модулю в сериализованном виде.

## История запусков в режиме `database`

Сущности ниже проектируются только для режима `database`.

Это означает:

- в PostgreSQL храним историю запусков только DB-модулей;
- для режима `files` сохраняем существующий механизм истории запусков, который уже реализован в UI;
- на первом этапе не объединяем file-history и database-history в одну общую модель хранения.

## Компонент очистки истории запусков в режиме `database`

Для режима `database` нужен отдельный компонент очистки истории запусков всех модулей.

Предварительное рабочее название:

- `ModuleRunHistoryCleanupService`

Назначение:

- удалить из PostgreSQL историю запусков всех модулей старше указанной даты;
- удалить связанные execution snapshot и зависимые данные, которые больше не нужны;
- при необходимости удалить файлы артефактов на диске, если они еще существуют.

Это именно отдельный служебный компонент режима `database`, а не общая логика для `files` и `database`.

Отдельно фиксируем:

- cleanup является общей операцией по всем DB-модулям сразу;
- cleanup не запускается "для одного модуля";
- cleanup не должен жить внутри страницы истории конкретного модуля.

### Что должен уметь компонент

Минимальная операция:

- `cleanupOlderThan(cutoffDateTime)`

Поведение:

- если дата передана явно, очищаются все данные старше нее;
- если дата не передана, по умолчанию используется текущий момент минус один календарный месяц.
- по умолчанию cleanup должен сохранять минимум `30` запусков на каждый модуль;
- это ограничение должно быть опциональным и может быть отключено явным флагом.

Предварительные параметры операции:

- `cutoffDateTime`
- `preserveMinRunsPerModule = true`
- `minRunsPerModule = 30`

### Какие данные считаются историей запуска

Очистка должна затрагивать только run-history сущности режима `database`:

- `module_run`
- `module_run_source_result`
- `module_run_event`
- `module_run_artifact`
- `execution_snapshot`

Не затрагиваются:

- `module`
- `module_revision`
- `module_working_copy`
- `module_sync_run`
- `module_sync_run_item`

### По какому времени считать запись устаревшей

Правильное правило:

- для завершенных запусков возраст определяем по `finished_at`;
- если по какой-то причине `finished_at` отсутствует, fallback идет на `requested_at`;
- незавершенные запуски (`RUNNING`, `REQUESTED`) компонент удалять не должен.

Итоговый критерий можно формулировать так:

- очищаем только терминальные запуски;
- терминальный запуск подлежит удалению, если его эффективное время завершения раньше `cutoffDateTime`.
- при этом, если включен `preserveMinRunsPerModule`, cleanup не должен уменьшать число запусков конкретного модуля ниже заданного минимума.

Это означает:

- если у модуля всего `30` запусков или меньше, по умолчанию его история не очищается вообще;
- если у модуля запусков больше `30`, cleanup может удалить только ту часть старой истории, которая не нарушает этот нижний порог;
- если пользователь отключил `preserveMinRunsPerModule`, cleanup может очищать историю модуля и ниже этого порога.

### Что значит "очистить запуск"

Для одного устаревшего запуска компонент должен:

1. Найти связанные `module_run_artifact`.
2. Попытаться удалить физические файлы артефактов на диске, если они существуют.
3. Удалить запись `module_run`.
4. За счет каскадных связей удалить:
   - `module_run_source_result`
   - `module_run_event`
   - `module_run_artifact`
5. После удаления запусков удалить orphan `execution_snapshot`, на которые больше не ссылается ни один `module_run`.

### Правила удаления файлов артефактов

Считаем правильным такое поведение:

- если файл артефакта существует, компонент пытается его удалить;
- если файл уже удален, это не считается ошибкой cleanup;
- если удалить файл не удалось, это должно фиксироваться в отчете cleanup, но не должно останавливать очистку остальных запусков.

Это нужно для того, чтобы cleanup не зависел от единичных проблем с файловой системой.

### Что должен возвращать cleanup

Компонент должен возвращать структурированный результат операции:

- `cutoffDateTime`
- `preserveMinRunsPerModule`
- `minRunsPerModule`
- `deletedRunCount`
- `deletedExecutionSnapshotCount`
- `deletedArtifactFileCount`
- `failedArtifactFileDeleteCount`
- `skippedRunningRunCount`
- `protectedModuleCount`
- `protectedRunCount`
- `errors`

Это нужно для:

- логирования;
- диагностики;
- возможного показа результата в UI или admin-операциях.

### Поведение по умолчанию

На текущем этапе фиксируем такое default-поведение:

- cleanup работает по всем модулям сразу;
- если дата не указана, чистим историю старше одного календарного месяца;
- по умолчанию сохраняем минимум `30` запусков на каждый модуль;
- cleanup удаляет и записи из БД, и файлы артефактов на диске;
- cleanup не трогает незавершенные запуски;
- cleanup не трогает ревизии, working copy и sync-историю.

### Как это соотносится с retention

На текущем этапе считаем это не "автоматической retention-политикой БД", а явной служебной операцией.

То есть сначала планируем:

- отдельный cleanup-компонент;
- явный вызов cleanup по команде или из служебного workflow.

Автоматическое расписание очистки можно добавить позже, если реально потребуется.

### 11. Сущность `module_run`

Назначение:

- одна запись на один фактический запуск модуля;
- основная сущность истории запусков в режиме `database`;
- источник данных для списка запусков, карточки текущего запуска и summary.

Описание полей:

- `run_id` — внутренний уникальный идентификатор запуска.
- `module_id` — ссылка на модуль, который был запущен.
- `execution_snapshot_id` — ссылка на точный snapshot конфигурации, с которым реально выполнялся запуск.
- `requested_by_actor_id` — идентификатор пользователя, который инициировал запуск.
- `requested_by_actor_source` — источник идентичности инициатора запуска.
- `requested_by_actor_display_name` — отображаемое имя инициатора запуска.
- `requested_at` — дата и время, когда пользователь запросил запуск.
- `started_at` — дата и время фактического старта выполнения.
- `finished_at` — дата и время завершения выполнения.
- `status` — итоговый статус запуска.
- `launch_source_kind` — из чего был создан execution snapshot: `CURRENT_REVISION` или `WORKING_COPY`.
- `module_code_snapshot` — код модуля на момент запуска; нужен для стабильного отображения истории, даже если модуль потом переименуют.
- `module_title_snapshot` — отображаемое имя модуля на момент запуска.
- `output_dir` — каталог output, который использовался этим запуском.
- `merge_mode` — режим merge, использованный в запуске.
- `merged_row_count` — итоговое количество строк в merged result, если merge был выполнен.
- `successful_source_count` — сколько источников завершились успешно.
- `failed_source_count` — сколько источников завершились ошибкой.
- `skipped_source_count` — сколько источников были пропущены или исключены из merge.
- `target_enabled` — был ли target включен в конфигурации запуска.
- `target_status` — итоговый статус target-этапа.
- `target_table_name` — имя target-таблицы, если target был включен.
- `target_rows_loaded` — число строк, реально загруженных в target, если оно известно.
- `summary_json` — денормализованный summary этого запуска для быстрого UI-рендеринга и сохранности информации при удалении output-файлов.
- `error_message` — короткое итоговое сообщение об ошибке, если запуск завершился неуспешно.

Важно:

- `module_run` не заменяет `execution_snapshot`, а ссылается на него;
- `module_run` хранит denormalized срез ключевых результатов, чтобы UI не собирал историю из пяти таблиц при каждом открытии страницы.

### 12. Сущность `module_run_source_result`

Назначение:

- результат выполнения одного source внутри конкретного запуска;
- источник данных для блока `Состояние источников` и source-level summary.

Описание полей:

- `run_source_result_id` — внутренний уникальный идентификатор записи.
- `run_id` — ссылка на запуск, к которому относится результат.
- `source_name` — имя источника в рамках этого запуска.
- `sort_order` — порядок отображения source в UI.
- `status` — итоговый статус выполнения source.
- `started_at` — время старта обработки source.
- `finished_at` — время завершения обработки source.
- `exported_row_count` — сколько строк реально выгружено из source.
- `merged_row_count` — сколько строк этого source попало в merged result.
- `error_message` — текст ошибки source, если source завершился неуспешно.

Важно:

- `exported_row_count` и `merged_row_count` не обязательно равны;
- это позволяет хранить и source export, и итоговое участие source в merge.

### 13. Сущность `module_run_event`

Назначение:

- append-only лента событий запуска;
- источник данных для блока `Ход выполнения`.

Описание полей:

- `run_event_id` — внутренний уникальный идентификатор события.
- `run_id` — ссылка на запуск, к которому относится событие.
- `seq_no` — монотонный номер события внутри одного запуска.
- `created_at` — время фиксации события.
- `stage` — этап выполнения, к которому относится событие.
- `event_type` — тип события.
- `severity` — важность события для UI.
- `source_name` — имя источника, если событие связано с конкретным source.
- `message` — человекочитаемый текст события для UI.
- `payload_json` — структурированные технические детали события.

Важно:

- timeline UI должен опираться именно на `module_run_event`, а не восстанавливаться по логам;
- `payload_json` позволяет не зашивать все частные поля в схему.

### 14. Сущность `module_run_artifact`

Назначение:

- учет файлов и других материальных артефактов запуска.

Описание полей:

- `run_artifact_id` — внутренний уникальный идентификатор артефакта.
- `run_id` — ссылка на запуск, к которому относится артефакт.
- `artifact_kind` — тип артефакта: source output, merged output, summary и т.д.
- `artifact_key` — ключ артефакта внутри запуска; например имя source или `merged`.
- `file_path` — путь к артефакту на диске.
- `storage_status` — текущее состояние артефакта на диске.
- `file_size_bytes` — размер файла, если он известен.
- `content_hash` — контрольная сумма артефакта, если она была вычислена.
- `created_at` — время регистрации артефакта.

Важно:

- история запусков не должна исчезать после retention output-файлов;
- если файл позже удален, запись в БД сохраняется, а `storage_status` меняется.

## Решение по `commonSql/commonSqlFile` и `source.sql/sqlFile`

Это зафиксировано как принципиальное решение:

- в БД не хотим делать два разных режима хранения SQL;
- в БД SQL должен храниться единообразно как текстовый SQL asset;
- различие между inline SQL и SQL из файла сохраняется только как metadata происхождения.

То есть:

- `commonSql` и `commonSqlFile` после загрузки в БД превращаются в один `COMMON` asset;
- `source.sql` и `source.sqlFile` после загрузки в БД превращаются в один `SOURCE` asset;
- `origin_kind` показывает, пришел SQL из inline-конфига или из файла;
- `origin_path` хранит исходный путь файла, если SQL был загружен из `commonSqlFile` или `sqlFile`.

Из этого следуют правила:

- runtime в режиме `database` не зависит от файлов;
- запуск работает только с `sql_text`, хранящимся в БД;
- `origin_kind` и `origin_path` сохраняются только как metadata происхождения SQL.

Таким образом:

- SQL в БД хранится канонически;
- формат файлового представления не теряется;
- но обратное восстановление модуля в файловую структуру не является частью текущей архитектуры.

## Правило по credentials

Это зафиксировано как отдельный принцип:

- PostgreSQL-реестр модулей не должен хранить реальные credentials;
- логины и пароли в БД не хранятся в resolved-виде;
- для `sources` и `target` в БД сохраняются только placeholder/expression значения;
- runtime должен получать реальные значения из `credential.properties`.

## Отдельная синхронизация `files -> database`

Синхронизацию не считаем частью обычного runtime-доступа к модулю.

Предварительно нужен отдельный сервис:

- `ModuleSyncService`

Операции:

- `syncAllFromFiles`
- `syncOneFromFiles`

То есть:

- обычная работа UI в режиме `database` идет через `DatabaseModuleStore`;
- загрузка модулей из `apps` в PostgreSQL идет через отдельный sync-сервис.

Это фиксирует границу текущего этапа:

- импорт выполняется только из файлов в БД;
- операций синхронизации из БД обратно в файловый модуль нет.

## Как должен работать `ModuleSyncService`

Рекомендуемый алгоритм:

1. Создать запись `module_sync_run` со статусом `RUNNING`.
2. Найти кандидаты в `appsRoot`.
3. Для каждого модуля:
   - быстро проверить изменения по `mtime` / `size`;
   - если precheck говорит, что модуль потенциально изменен, посчитать канонический `content_hash`;
   - если hash не изменился, зафиксировать `SKIPPED`;
   - если модуль новый или hash изменился:
     - прочитать `ui-module.yml`;
     - прочитать `application.yml`;
     - прочитать все реально используемые SQL-файлы;
     - собрать каноническую runtime-модель;
     - провалидировать содержимое;
     - создать новую `module_revision`;
     - переключить `module.current_revision_id` на нее.
4. Если у модуля есть `working copy` пользователей, и их `base_revision_id` больше не совпадает с новым `current_revision_id`, пометить такие `working copy` как `STALE`.
5. По каждому модулю создать запись `module_sync_run_item`.
6. Завершить `module_sync_run` итоговым статусом:
   - `SUCCESS`
   - `PARTIAL_SUCCESS`
   - `FAILED`

Что считаем правильным:

- sync не должен затирать `working copy`;
- sync не должен создавать `working copy`;
- sync не должен автоматически публиковать пользовательские drafts;
- sync не должен удалять модули из БД, которых больше нет в `apps`.

## Как `DatabaseModuleStore` должен работать с YAML

Ключевое решение:

- YAML в DB-режиме не должен собираться "на лету" из десятка таблиц при каждом чтении.

Правильный подход:

- при каждом sync, import или создании ревизии backend транзакционно обновляет:
  - структурированные таблицы ревизии;
  - `snapshot_json`;
  - `snapshot_yaml`.
- при каждом сохранении рабочей копии backend транзакционно обновляет:
  - `working_copy_json`;
  - `working_copy_yaml`;
  - `content_hash`.

Следствие:

- `EditableModule.configText` в режиме `database` возвращается из уже сохраненного канонического YAML:
  - `working_copy_yaml`, если есть рабочая копия;
  - иначе `snapshot_yaml` текущей ревизии.

Это снимает проблемы:

- дрейф формата YAML между чтениями;
- случайная перестановка полей из-за повторной сериализации;
- лишняя сложность в `DatabaseModuleStore`.

## Как должен жить `module_run`

Рекомендуемый жизненный цикл запуска:

1. Создать `execution_snapshot`.
2. Создать `module_run`:
   - `status = RUNNING`
   - заполнить `requested_at`
   - заполнить denormalized поля запуска, известные на старте.
3. Создать начальные записи `module_run_source_result` со статусом `PENDING`.
4. Во время выполнения:
   - обновлять `module_run_source_result`;
   - добавлять `module_run_event`;
   - регистрировать `module_run_artifact` по мере появления файлов.
5. По завершении:
   - обновить агрегированные поля в `module_run`;
   - записать `summary_json`;
   - поставить финальный `status`;
   - заполнить `finished_at`.

Отдельно фиксируем:

- если процесс UI/backend перезапустился и в БД остались запуски со статусом `RUNNING`, при старте приложения они должны быть переведены в `FAILED`;
- для таких запусков нужно:
  - заполнить `finished_at`;
  - записать `error_message` вида "Запуск прерван из-за рестарта процесса";
  - добавить финальное событие в `module_run_event`.

Это нужно для того, чтобы история не содержала "вечных" зависших запусков.

## Рекомендуемый DDL-подход

Для PostgreSQL считаем правильным:

- использовать отдельную схему, например `ui_registry`;
- использовать `uuid` как идентификаторы;
- использовать `timestamptz` для всех моментов времени;
- использовать `jsonb` для:
  - `tags`
  - `validation_issues`
  - `snapshot_json`
  - `working_copy_json`
  - `details`
- использовать `text + check constraint`, а не PostgreSQL enum types.

Минимально необходимые индексы:

- `module(module_code)` — уникальный поиск модуля по коду;
- `module(current_revision_id)` — переход к текущей ревизии;
- `module_revision(module_id, revision_no desc)` — быстрый доступ к истории ревизий;
- `module_revision(content_hash)` — сопоставление по содержимому;
- `module_revision_source(revision_id, source_name)` — источники ревизии;
- `module_revision_sql_asset(revision_id, asset_kind, asset_key)` — SQL-артефакты ревизии;
- `module_working_copy(module_id, owner_actor_id, owner_actor_source)` — персональная рабочая копия модуля;
- `execution_snapshot(module_id, created_at desc)` — история снапшотов запуска по модулю;
- `execution_snapshot(requested_by_actor_id, requested_by_actor_source, created_at desc)` — история запусков пользователя;
- `module_sync_run(started_at desc)` — история sync-запусков;
- `module_sync_run(started_by_actor_id, started_by_actor_source, started_at desc)` — история sync-запусков пользователя;
- `module_sync_run_item(sync_run_id, module_code)` — детализация sync по модулям.
- `module_run(module_id, requested_at desc)` — история запусков модуля;
- `module_run(requested_by_actor_id, requested_by_actor_source, requested_at desc)` — история запусков пользователя;
- `module_run(status, requested_at desc)` — быстрый доступ к активным и проблемным запускам;
- `module_run_source_result(run_id, source_name)` — результаты source внутри запуска;
- `module_run_event(run_id, seq_no)` — timeline запуска в правильном порядке;
- `module_run_artifact(run_id, artifact_kind, artifact_key)` — артефакты запуска.

## Черновой PostgreSQL DDL

Ниже фиксируем рекомендуемый SQL-каркас схемы. Это еще не миграция, но уже целевой DDL-level дизайн.

### Базовые решения

- используем схему `ui_registry`;
- все `uuid` генерируются приложением, а не базой данных;
- все `updated_at` поля на первом этапе обновляются приложением, а не trigger-ами БД;
- ревизия и execution snapshot считаются immutable;
- `module.current_revision_id` проверяем составным FK, чтобы текущая ревизия гарантированно принадлежала своему модулю.

### Порядок создания

Правильный порядок такой:

1. `module`
2. `module_revision`
3. `module_revision_sql_asset`
4. `module_revision_source`
5. `module_revision_target`
6. `module_revision_quota`
7. `module_sync_run`
8. `module_working_copy`
9. `execution_snapshot`
10. `module_sync_run_item`
11. `module_run`
12. `module_run_source_result`
13. `module_run_event`
14. `module_run_artifact`
15. финальный FK `module -> current_revision`

### SQL-каркас

```sql
create schema if not exists ui_registry;

create table ui_registry.module (
    module_id uuid primary key,
    module_code text not null unique,
    module_origin_kind text not null,
    current_revision_id uuid null,
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now(),
    constraint chk_module_code_not_blank check (btrim(module_code) <> ''),
    constraint chk_module_origin_kind check (module_origin_kind in ('IMPORTED_FROM_FILES', 'CREATED_IN_UI'))
);

create table ui_registry.module_revision (
    revision_id uuid primary key,
    module_id uuid not null references ui_registry.module(module_id) on delete cascade,
    revision_no bigint not null,
    created_at timestamptz not null default now(),
    created_by text null,
    revision_source text not null,
    title text not null,
    description text null,
    tags jsonb not null default '[]'::jsonb,
    hidden_from_ui boolean not null default false,
    validation_status text not null,
    validation_issues jsonb not null default '[]'::jsonb,
    output_dir text not null,
```

### Что важно в этом DDL

- база не зависит от PostgreSQL extensions для генерации идентификаторов;
- приложение обязано всегда передавать `uuid` при вставке записей;
- `module.current_revision_id` проверяется составным FK и не может указывать на ревизию другого модуля;
- `module_working_copy` тоже привязана к своей базовой ревизии через составной FK;
- `execution_snapshot` обязан ссылаться ровно на один источник:
  - либо на ревизию;
  - либо на working copy;
- `module_run` в этой модели относится только к `database` режиму;
- `module_run_event.payload_json` и `module_run.summary_json` позволяют развивать UI без постоянных миграций схемы;
- `module_run_artifact` не хранит бинарные данные файлов, а только их учет и статус на диске.

## Стратегия миграций

На текущем этапе считаем правильным следующее решение:

- использовать `Flyway` для всех изменений PostgreSQL-схемы режима `database`;
- первый rollout сделать одним versioned migration-скриптом;
- этот скрипт должен создавать:
  - схему `ui_registry`;
  - все таблицы;
  - все ограничения;
  - все индексы.

### Правило на первый этап

Пока модель еще проектируется и не начала жить в production, допустимо держать один baseline-скрипт:

- `V1__create_ui_registry.sql`

В нем должен быть весь DDL, зафиксированный в этом черновике.

### Правило после первого применения

После того как `V1` будет применен хотя бы в одной рабочей среде, считаем правильным:

- больше не переписывать `V1`;
- все изменения делать только новыми миграциями:
  - `V2__...`
  - `V3__...`
  - и так далее.

### Что считаем правильным для Flyway

- использовать versioned migrations, а не repeatable migrations, для схемы реестра модулей;
- не делать одну "вечную" migration, которая постоянно переписывается после первого реального применения;
- не смешивать миграции режима `database` с чем-то, что относится только к `files` режиму.

### Предварительное расположение

Когда дойдем до реализации, считаем правильным стандартный путь:

- `src/main/resources/db/migration/V1__create_ui_registry.sql`

Если позже появится отдельный модуль для DB-registry инфраструктуры, путь можно будет уточнить, но формат и подход должны остаться такими же.

## Где хранить выбранный режим `FILES / DB`

Решение:

- выбранный пользователем режим должен храниться в UI storage;
- основной UI-конфиг должен хранить только возможности среды:
  - `appsRoot`
  - настройки PostgreSQL для режима `database`

Стартовое поведение:

- при первом запуске по умолчанию открывается `FILES`;
- при последующих запусках UI пытается восстановить последний выбранный режим;
- если был сохранен `DB`, но PostgreSQL недоступен, UI принудительно открывает `FILES`.

## UX для `Publish / STALE / conflict` в режиме `database`

Этот UX относится только к режиму `database`.

Для режима `files` такие состояния и операции не вводим.

### Базовый принцип UX

Пользователь должен всегда явно понимать:

- работает ли он с опубликованной ревизией или со своей личной `working copy`;
- есть ли у него неопубликованные изменения;
- устарела ли его `working copy` относительно текущей опубликованной ревизии;
- что именно произойдет при `Save`, `Publish`, `Run` и `Discard`.

UI не должен маскировать эти различия.

### Верхняя статусная панель модуля в режиме `database`

В карточке редактирования модуля нужен отдельный статусный блок.

В нем должны отображаться:

- текущий режим: `DB`;
- код и имя модуля;
- текущая опубликованная ревизия:
  - номер;
  - кем опубликована;
  - когда опубликована;
- состояние личной `working copy` текущего пользователя;
- если `working copy` существует:
  - ее базовая ревизия;
  - время последнего изменения.

Это должен быть обязательный элемент экрана, а не второстепенная диагностика.

### Основные действия пользователя

В режиме `database` на экране модуля должны быть явные действия:

- `Save`
- `Publish`
- `Discard working copy`

Дополнительные действия, которые считаем правильными:

- `Reload current revision`
- `Run`

Смысл действий:

- `Save`:
  - сохраняет только личную `working copy` текущего пользователя;
  - не меняет опубликованную ревизию;
- `Publish`:
  - создает новую общую ревизию;
  - меняет `current_revision_id`;
- `Discard working copy`:
  - удаляет личную рабочую копию пользователя;
  - возвращает экран к опубликованной ревизии;
- `Reload current revision`:
  - перечитывает текущее опубликованное состояние модуля;
  - нужен для ручного обновления экрана после конфликтов и sync.

### Состояния экрана

Считаем правильным выделять четыре основных состояния.

#### 1. Нет `working copy`

Это означает:

- пользователь сейчас смотрит опубликованную ревизию;
- личной рабочей копии у него нет.

UI-поведение:

- status banner нейтральный;
- `Save` создает новую `working copy`;
- `Publish` disabled;
- `Discard working copy` hidden или disabled;
- `Run` запускает текущую опубликованную ревизию.

Текстовое сообщение должно быть примерно такого смысла:

- `Личной рабочей копии нет. Вы работаете с опубликованной ревизией.`

#### 2. `working copy` со статусом `CLEAN`

Это означает:

- личная рабочая копия существует;
- она все еще основана на актуальной `current revision`;
- содержательно она не отличается от базовой ревизии.

UI-поведение:

- banner нейтральный или informational;
- `Save` доступен;
- `Publish` disabled, потому что публиковать нечего;
- `Discard working copy` доступен;
- `Run` запускает личную `working copy`, хотя фактически она совпадает с опубликованной ревизией.

Текстовое сообщение:

- `У вас есть личная рабочая копия без неопубликованных изменений.`

#### 3. `working copy` со статусом `DIRTY`

Это означает:

- личная рабочая копия существует;
- она основана на актуальной `current revision`;
- в ней есть неопубликованные изменения.

UI-поведение:

- banner warning/in-progress;
- `Save` доступен;
- `Publish` доступен, если модуль проходит валидацию;
- `Discard working copy` доступен;
- `Run` запускает личную `working copy`;
- на кнопке запуска или рядом с ней должно быть понятно, что будет запущена именно рабочая копия пользователя, а не опубликованная ревизия.

Текстовое сообщение:

- `У вас есть неопубликованные изменения. Они видны только вам, пока вы не выполните Publish.`

#### 4. `working copy` со статусом `STALE`

Это означает:

- личная рабочая копия основана на старой ревизии;
- с момента ее создания `current revision` успела измениться через sync или publish другого пользователя.

UI-поведение:

- banner warning/error;
- `Publish` жестко disabled;
- `Save` остается доступным, потому что пользователь не должен терять свой draft;
- `Discard working copy` доступен;
- `Run` не выполняется сразу, а требует явного подтверждения;
- UI обязан явно показать, что личная копия устарела относительно опубликованной ревизии.

Текстовое сообщение:

- `Ваша рабочая копия устарела. С момента ее создания опубликованная ревизия изменилась.`

### Почему `Save` не блокируем для `STALE`

Считаем правильным не запрещать `Save` для устаревшей рабочей копии.

Причины:

- пользователь не должен терять свои изменения;
- иногда ему нужно еще сохранить текущее draft-состояние перед тем, как отбросить или пересоздать рабочую копию;
- блокировать нужно только опасные действия:
  - `Publish`
  - автоматический `Run`

### Поведение `Publish`

`Publish` должен работать только при выполнении всех условий:

- режим `database`;
- у пользователя есть `working copy`;
- статус `working copy` = `DIRTY`;
- модуль проходит валидацию и не имеет блокирующих ошибок;
- `working_copy.base_revision_id == current_revision_id`.

Если все условия соблюдены:

- backend создает новую `module_revision`;
- backend переключает `module.current_revision_id`;
- backend удаляет личную `working copy` как уже опубликованную;
- UI после успешного ответа перечитывает модуль и показывает опубликованную ревизию без `working copy`.

Если условия не соблюдены:

- publish не должен происходить частично;
- backend возвращает детальную причину отказа;
- UI отображает причину пользователю в явном виде.

### Как должен выглядеть optimistic conflict при `Publish`

Если во время публикации выяснилось, что:

- `working_copy.base_revision_id != current_revision_id`

то backend должен вернуть конфликт публикации, а не generic error.

Правильная реакция backend:

- HTTP `409 Conflict`;
- рабочая копия пользователя помечается как `STALE`;
- в ответе возвращаются:
  - `moduleId`
  - `workingCopyId`
  - `workingCopyStatus = STALE`
  - `baseRevisionId`
  - `currentRevisionId`
  - `currentRevisionNo`
  - `currentRevisionPublishedAt`
  - `currentRevisionPublishedBy`
  - `message`

Правильная реакция UI:

- перечитать состояние модуля;
- показать warning banner;
- отключить `Publish`;
- предложить безопасные действия вместо повторной публикации.

### Какие действия должен предлагать UI при `STALE`

На первом этапе считаем достаточными такие действия:

- `Discard working copy`
- `Reload current revision`

Смысл:

- `Discard working copy`:
  - удаляет устаревший личный draft;
  - после этого пользователь работает с актуальной опубликованной ревизией;
- `Reload current revision`:
  - просто обновляет экран и перечитывает актуальное состояние из БД.

Что сознательно не делаем на первом этапе:

- автоматический merge;
- rebase draft на новую ревизию;
- полноценный визуальный diff;
- сложный conflict resolution workflow.

### Поведение `Run`

Правильное правило:

- если `working copy` отсутствует, запускается `current revision`;
- если `working copy` имеет статус `CLEAN` или `DIRTY`, запускается `working copy`;
- если `working copy` имеет статус `STALE`, автоматический запуск запрещен.

Для `STALE` UI должен показать confirm-диалог.

Правильные действия в таком диалоге:

- `Отмена`
- `Запустить устаревшую working copy`
- `Discard working copy и работать с current revision`

Принцип:

- запуск `STALE`-копии допускается только как осознанное действие пользователя;
- по умолчанию UI не должен выбирать этот путь автоматически.

### Поведение после успешного `Publish`

После успешной публикации UI должен:

1. Показать короткое success-сообщение.
2. Перечитать состояние модуля из backend.
3. Обновить статусную панель:
   - новая `current revision`;
   - личная `working copy` отсутствует.
4. Обновить список модулей, если там показывается номер ревизии или время публикации.

### Что должен возвращать backend на `Publish`

Помимо HTTP-статуса, backend должен возвращать структурированный ответ:

- `moduleId`
- `publishedRevisionId`
- `publishedRevisionNo`
- `publishedAt`
- `publishedByActorId`
- `publishedByActorDisplayName`
- `workingCopyRemoved`
- `message`

Это нужно, чтобы UI не гадал, что произошло, а мог стабильно перестроить экран.

### Что считаем неправильным для UX

Не хотим:

- молча запускать устаревшую `working copy`;
- молча затирать draft пользователя после sync или чужого publish;
- давать `Publish`, который иногда работает по optimistic concurrency, а иногда silently replaces current revision;
- прятать различие между опубликованной ревизией и личной рабочей копией;
- показывать generic `500` вместо отдельного publish-conflict сценария.

Итоговый принцип:

- `Save` — только про личный draft;
- `Publish` — только про общую ревизию;
- `STALE` — всегда явное проблемное состояние, а не техническая деталь.

## Импорт из файлов в БД

На текущем этапе фиксируем только один поддерживаемый поток:

- `files -> database`

Это означает:

- источником модулей для загрузки в БД являются только файловые модули из `appsRoot`;
- операции синхронизации и импорта работают только в сторону PostgreSQL;
- обратный поток из БД в файловый модуль не поддерживается.

### Что именно импортируется

При импорте одного модуля из файлов в БД должны переноситься:

- `ui-module.yml`
- `application.yml`
- все реально используемые SQL-файлы модуля
- вычисленные metadata:
  - `content_hash`
  - `validation_status`
  - `validation_issues`

### Какие операции импорта поддерживаются

Поддерживаются только две операции:

- полный импорт:
  - `syncAllFromFiles`
- точечный импорт:
  - `syncOneFromFiles`

Других сценариев импорта на текущем этапе не закладываем.

### Что не считаем импортом на текущем этапе

Не проектируем и не реализуем:

- импорт модуля из ZIP/архива;
- импорт из произвольного YAML/JSON вне `appsRoot`;
- импорт из другой БД;
- двустороннюю синхронизацию файлов и БД.

### Что будет считаться успешным импортом

Импорт модуля считаем успешным, если:

- модуль найден в `appsRoot`;
- `ui-module.yml` и `application.yml` успешно прочитаны;
- все реально используемые SQL-артефакты успешно прочитаны;
- вычислен `content_hash`;
- создана новая ревизия или корректно зафиксирован `SKIPPED`, если содержимое не изменилось;
- обновлены записи `module_sync_run` и `module_sync_run_item`.

Правило по происхождению модуля:

- если в ходе import модуль впервые создается в БД, для него фиксируется:
  - `module_origin_kind = IMPORTED_FROM_FILES`
- если модуль уже существует в БД, его `module_origin_kind` не изменяется.

### Что делать при конфликте `moduleCode`

Считаем правильным не перезаписывать существующий DB-модуль при совпадении кода.

Правило:

- если `sync from files` находит файловый модуль с `moduleCode`, который уже существует в БД, import такого модуля не выполняется;
- существующий DB-модуль не изменяется;
- новая ревизия не создается;
- в `module_sync_run_item` фиксируется отдельный конфликтный результат;
- пользователю показывается рекомендация переименовать файловый или DB-модуль.

Обоснование:

- это проще и безопаснее, чем "перехватывать" существующий DB-модуль импортом;
- не возникает скрытого изменения `current_revision`;
- не переводятся неожиданно `working copy` пользователей в `STALE`;
- правило остается однозначным и понятным.

### Принципиальная граница текущего этапа

Текущий этап не должен превращаться в двустороннюю систему синхронизации.

Фиксируем:

- БД наполняется из файлов;
- файлы не обновляются из БД;
- поток `database -> files` не рассматривается.

## UX импорта из файлов в БД

Этот UX относится только к режиму `database`.

Для режима `files` никаких операций импорта и sync в UI не должно быть.

### Где размещать действия импорта

Считаем правильным не перегружать основную страницу модулей действиями sync.

Поэтому импорт должен быть вынесен на отдельную страницу.

Предварительное название страницы:

- `Импорт моделей в БД`

Со страницы модулей должен быть переход на эту страницу через отдельную кнопку:

- `Импорт моделей в БД`

На странице модулей эта кнопка показывается только в режиме `database`.

Это важно, потому что sync относится к наполнению DB-каталога и служебным операциям, а не к повседневному редактированию конкретного модуля.

### Какие действия должны быть в UI

На первом этапе нужны только две явные кнопки:

- `Обновить все модули в БД`
- `Обновить выбранный модуль`

Правила:

- `Обновить все модули в БД` работает по всему `appsRoot`;
- `Обновить выбранный модуль` доступна только если на странице импорта выбран конкретный файловый модуль;
- никаких дополнительных режимов импорта не добавляем.

Дополнительно на странице импорта должна быть кнопка возврата:

- `К модулям`

### Предварительные проверки перед импортом

Перед запуском sync UI и backend должны проверять:

- активен ли режим `database`;
- доступна ли PostgreSQL;
- задан ли `appsRoot`;
- существует ли `appsRoot` и является ли директорией;
- для точечного sync:
  - выбран ли модуль на странице импорта;
  - существует ли соответствующий файловый модуль в `appsRoot`.

Если одна из проверок не проходит:

- sync не стартует;
- пользователь получает понятную ошибку;
- никаких частичных действий в БД не выполняется.

### Как показывать состояние `appsRoot`

На странице импорта нужен отдельный компактный статус-блок по `appsRoot`.

Он должен показывать:

- путь `appsRoot`;
- валиден ли он;
- сколько файловых модулей найдено;
- можно ли выполнять import/sync.

Статусы:

- `appsRoot доступен`
- `appsRoot не задан`
- `appsRoot не найден`
- `appsRoot недоступен`

Если `appsRoot` невалиден:

- обе кнопки sync disabled;
- рядом должен быть текст, почему импорт невозможен.

### Что должно быть на странице импорта

Минимальный состав страницы:

1. заголовок `Импорт моделей в БД`
2. кнопка `К модулям`
3. статус подключения к PostgreSQL
4. статус `appsRoot`
5. кнопка `Обновить все модули в БД`
6. список файловых модулей из `appsRoot`
7. кнопка `Обновить выбранный модуль`
8. блок результата последнего sync

Это должна быть отдельная служебная страница, а не модальное окно и не встроенный side-panel.

### Полный импорт

Flow для `Обновить все модули в БД`:

1. Пользователь нажимает кнопку.
2. UI показывает confirm с понятным смыслом операции:
   - будут проверены все модули из `appsRoot`;
   - новые и измененные модули попадут в БД;
   - удаленные из файлов модули из БД удаляться не будут.
3. После подтверждения UI запускает `module_sync_run`.
4. Во время выполнения:
   - кнопки sync disabled;
   - показывается статус `Импорт выполняется`;
   - UI может периодически обновлять состояние sync.
5. После завершения:
   - UI показывает итоговую сводку;
   - список модулей перечитывается из БД.

### Точечный импорт

Flow для `Обновить выбранный модуль`:

1. Пользователь переходит на страницу `Импорт моделей в БД`.
2. Выбирает файловый модуль в списке.
3. Нажимает `Обновить выбранный модуль`.
4. UI запускает `syncOneFromFiles(moduleCode)`.
5. Во время выполнения:
   - кнопка `Обновить выбранный модуль` disabled;
   - глобальный full-sync можно тоже блокировать, чтобы не запускать конкурирующие sync-операции.
6. После завершения:
   - перечитывается состояние выбранного файлового модуля на странице импорта;
   - после возврата на страницу модулей UI читает уже обновленный DB-каталог.

### Что должен видеть пользователь во время sync

На первом этапе достаточно простого статуса операции без сложного live-log.

Нужно показывать:

- что sync запущен;
- кто его запустил;
- когда он стартовал;
- что синхронизируется:
  - все модули
  - или конкретный модуль;
- общий статус:
  - `RUNNING`
  - `SUCCESS`
  - `PARTIAL_SUCCESS`
  - `FAILED`

Желательно также показать счетчики:

- сколько модулей обработано;
- сколько:
  - создано;
  - обновлено;
  - пропущено;
  - завершилось ошибкой.

### Как показывать результат sync

После завершения sync нужен явный result block.

Минимальный состав:

- итоговый статус операции;
- `startedAt`
- `finishedAt`
- `duration`
- `processedModuleCount`
- `createdCount`
- `updatedCount`
- `skippedCount`
- `failedCount`

Для full sync также нужен список проблемных модулей:

- `moduleCode`
- краткая ошибка

Для точечного sync достаточно показать:

- модуль обновлен;
- модуль не изменился;
- модуль завершился ошибкой.

### Как трактовать `SKIPPED`

`SKIPPED` не считаем ошибкой.

Для пользователя это должно читаться как:

- `Модуль не изменился, обновление не требовалось.`

То есть `SKIPPED` — это нормальный успешный результат конкретного модуля внутри sync.

### Что делать, если sync частично успешен

`PARTIAL_SUCCESS` должен считаться отдельным нормальным итоговым состоянием.

Это означает:

- часть модулей успешно импортирована;
- часть завершилась ошибкой;
- уже импортированные модули не откатываются только из-за того, что другие модули упали.

UI должен:

- показывать `PARTIAL_SUCCESS` как warning, а не как полный failure;
- давать пользователю список модулей с ошибками;
- позволять сразу повторить точечный import проблемного модуля.

Если модуль пропущен из-за конфликта кода, UI должен показывать отдельное понятное сообщение:

- `Импорт модуля <moduleCode> пропущен: в БД уже существует модуль с таким кодом. Переименуйте файловый или DB-модуль.`

### Что делать, если sync уже выполняется

На текущем этапе считаем правильным разделять два случая:

Правило:

- `syncAllFromFiles` — глобальная операция;
- `syncOneFromFiles` — операция уровня одного модуля.

Из этого следуют правила:

- одновременно может выполняться только один `syncAllFromFiles` на весь DB-режим;
- `syncAllFromFiles` несовместим с любыми `syncOneFromFiles`;
- `syncOneFromFiles` может выполняться параллельно с другими `syncOneFromFiles`, если это разные модули;
- два `syncOneFromFiles` для одного и того же `moduleCode` одновременно выполняться не должны;
- backend должен обеспечивать это правило сам и не полагаться только на UI.

Пользователю нужно показать понятное сообщение:

- если конфликт касается того же модуля:
  - `Импорт этого модуля уже выполняется. Дождитесь завершения текущей операции.`
- если активен полный импорт:
  - `Работа пока невозможна: идет массовый импорт модулей в БД.`

Дополнительное правило для полного импорта:

- если выполняется `syncAllFromFiles`, считаем, что DB-каталог находится в режиме служебного обновления;
- в этот момент работа остальных пользователей в режиме `database` временно недоступна;
- пользователь должен видеть явное сообщение:
  - `Работа пока невозможна: идет массовый импорт модулей в БД.`

### Многопользовательский сценарий одновременного импорта

Считаем правильным использовать двухуровневую блокировку импорта на уровне PostgreSQL.

Рекомендуемый механизм:

- один глобальный advisory lock для `syncAllFromFiles`;
- отдельный advisory lock на `moduleCode` для `syncOneFromFiles`;
- полный импорт может стартовать только если:
  - не удерживается глобальный full-sync lock;
  - не удерживается ни один module-level lock;
- точечный импорт может стартовать только если:
  - не удерживается глобальный full-sync lock;
  - не удерживается lock на этот же `moduleCode`.

Почему именно так:

- `module_sync_run.status = RUNNING` сам по себе не защищает от гонки между двумя почти одновременными запросами;
- UI может работать из нескольких клиентских сессий и даже из нескольких инстансов приложения;
- нужен backend-authoritative механизм взаимного исключения;
- при этом нет смысла запрещать параллельный импорт разных модулей, если они не пересекаются.

### Что происходит, если два пользователя запускают import одновременно

Нужны разные сценарии.

#### Сценарий 1. Два пользователя запускают import одного и того же модуля

1. Первый пользователь получает module-level lock для `moduleCode = X`.
2. Backend создает `module_sync_run` со статусом `RUNNING`.
3. Второй пользователь пытается запустить import этого же модуля `X`.
4. Backend не получает lock и отклоняет запуск.
5. UI второго пользователя показывает, что импорт этого модуля уже выполняется.

#### Сценарий 2. Два пользователя запускают import разных модулей

1. Первый пользователь запускает import модуля `X`.
2. Второй пользователь запускает import модуля `Y`.
3. Backend получает разные module-level lock.
4. Оба import могут выполняться параллельно.

#### Сценарий 3. Один пользователь запускает full sync, второй — точечный import

1. Если full sync уже запущен, точечный import не стартует.
2. Если точечный import уже запущен, full sync не стартует.
3. UI в обоих случаях показывает, что сейчас идет массовый импорт или активен другой import.

То есть:

- full sync эксклюзивен для всего DB-режима;
- точечные import могут идти параллельно только для разных модулей.

### Что должен видеть второй пользователь

Если import уже выполняется, на странице `Импорт моделей в БД` нужно показывать не просто ошибку, а состояние активной операции.

Минимально:

- статус `Импорт уже выполняется`
- кто запустил импорт
- когда он начался
- что выполняется:
  - полный импорт
  - или импорт конкретного модуля

В этом состоянии:

- `Обновить все модули в БД` disabled, если идет любой активный import;
- `Обновить выбранный модуль`:
  - disabled для модуля, который уже импортируется;
  - остается доступной для другого модуля, если нет full sync;
- страница может периодически обновлять статус текущего `module_sync_run`;
- после завершения активного sync кнопки снова становятся доступными.

Если активная операция — это `syncAllFromFiles`, дополнительное поведение должно быть таким:

- страница модулей в режиме `database` тоже показывает blocking banner;
- открытие и редактирование DB-модулей временно недоступно;
- запуск DB-модулей временно недоступен;
- пользователь может только дождаться завершения массового импорта или переключиться в `FILES`.

То есть full sync трактуем как кратковременный maintenance mode для DB-режима.

Если активная операция — это `syncOneFromFiles(moduleCode = X)`, дополнительное поведение должно быть таким:

- DB-режим в целом остается рабочим;
- модуль `X` на странице модулей помечается как временно недоступный;
- редактирование и запуск модуля `X` временно недоступны;
- другие DB-модули продолжают работать в обычном режиме.

### Что должен возвращать backend при конфликте запуска import

Если import уже выполняется, backend должен возвращать не generic error, а отдельный конфликт служебной операции.

Правильная реакция backend:

- HTTP `409 Conflict` или `423 Locked`;
- структурированный ответ:
  - `activeSyncRunId`
- `startedAt`
- `startedByActorId`
- `startedByActorDisplayName`
- `scope`
- `moduleCode`
  - `message`

Правильная реакция UI:

- показать активную операцию;
- не создавать локальное ощущение, что "кнопка просто не сработала";
- дать пользователю понять, кто именно сейчас выполняет import.

Если конфликт вызван именно `syncAllFromFiles`, UI должен показывать более жесткое сообщение:

- `Работа пока невозможна: идет массовый импорт модулей в БД.`

И в этом состоянии:

- нельзя открывать DB-модули для редактирования;
- нельзя запускать DB-модули;
- нельзя стартовать другой import;
- можно только ждать завершения операции или переключиться в `FILES`.

Если конфликт вызван `syncOneFromFiles` для того же модуля, UI должен показывать более точное сообщение:

- `Импорт модуля <moduleCode> уже выполняется другим пользователем.`

И в этом состоянии:

- нельзя редактировать и запускать именно этот модуль;
- можно продолжать работать с другими модулями;
- можно запускать точечный import для другого модуля, если нет активного full sync.

### Что делать при аварийном завершении import

Так как import может быть запущен несколькими пользователями и даже из разных инстансов UI, важно правильно обрабатывать аварийные остановки.

Правильный принцип:

- глобальная блокировка должна освобождаться автоматически при разрыве DB-сессии;
- если в БД осталась запись `module_sync_run` со статусом `RUNNING`, но lock уже свободен, следующий backend-запуск import должен:
  - пометить такую старую операцию как `FAILED`;
  - записать в `details`, что import был прерван;
  - только после этого создавать новый `module_sync_run`.

Это нужно для того, чтобы:

- UI не показывал "вечный RUNNING";
- новый import не упирался в исторический мусор;
- журнал sync оставался правдивым.

### Как sync влияет на открытый модуль

Если в UI открыт модуль, и sync обновил его `current revision`, экран должен корректно перестроиться.

Правильное поведение:

- если у пользователя нет `working copy`, экран после завершения sync показывает новую опубликованную ревизию;
- если у пользователя есть `working copy`, и она стала `STALE`, UI обязан явно показать это состояние;
- sync не должен молча затирать открытый редактор.

### Что считаем неправильным в UX импорта

Не хотим:

- прятать sync внутри карточки модуля;
- смешивать экран редактирования модуля и экран импорта;
- запускать sync автоматически без явного действия пользователя;
- автоматически удалять модули из БД, которых больше нет в `apps`;
- показывать `SKIPPED` как ошибку;
- молча перезаписывать существующий DB-модуль файловым import при совпадении `moduleCode`;
- разрешать второй sync, пока первый еще выполняется;
- silently refresh открытый модуль так, что пользователь не понимает, что `current revision` уже изменилась.

## API-контракты backend для режима `database`

Для DB-режима считаем правильным выделить отдельные backend API, а не переиспользовать файловые endpoints.

Базовый принцип:

- `files` и `database` имеют разные API-поверхности;
- DB-режим не должен маскироваться под текущие file-endpoints;
- backend-контракты должны отражать реальные сущности DB-режима:
  - `working copy`
  - `publish`
  - `sync`
  - `module run history`
  - `cleanup`

### Пространства API

На текущем этапе считаем правильным четыре группы endpoint'ов:

- `/api/db-mode/*`
- `/api/db-modules/*`
- `/api/db-import/*`
- `/api/db-runs/*`

Это дает:

- изоляцию по ответственности;
- меньше соблазна смешивать разные экраны и сценарии;
- более прозрачную серверную структуру.

### Общие правила для DB API

Все DB-endpoint'ы должны работать только при выполнении базовых условий:

- PostgreSQL доступна;
- actor определен автоматически или введен вручную;
- backend может работать в DB-режиме.

Если базовые условия не выполнены:

- endpoint не должен возвращать generic `500`;
- должен возвращаться управляемый бизнес-ответ с понятной причиной.

Предварительные HTTP-правила:

- `200 OK` — успешное чтение или успешная синхронная операция;
- `201 Created` — успешное создание модуля;
- `202 Accepted` — асинхронно запущенный процесс:
  - sync
  - module run;
- `400 Bad Request` — некорректный ввод;
- `404 Not Found` — модуль, run или sync-run не найден;
- `409 Conflict` — конфликт состояния:
  - `STALE`
  - активный sync
  - активный import lock
  - попытка удалить/опубликовать ресурс в неподходящем состоянии.

### Общий формат ошибок

На текущем этапе считаем правильным единый error payload:

- `errorCode`
- `message`
- `details`

Примеры `errorCode`:

- `DATABASE_MODE_UNAVAILABLE`
- `POSTGRES_UNAVAILABLE`
- `ACTOR_ID_REQUIRED`
- `WORKING_COPY_STALE`
- `SYNC_ALREADY_RUNNING`
- `MODULE_SYNC_ALREADY_RUNNING`
- `IMPORT_CODE_CONFLICT`
- `MODULE_DELETE_FORBIDDEN_ACTIVE_SYNC`
- `MODULE_DELETE_FORBIDDEN_ACTIVE_RUN`

`details` нужен для машиночитаемых полей UI.

## UX истории запусков для режима `database`

История запусков DB-модулей должна быть вынесена в отдельный экран.

Не хотим смешивать:

- редактирование модуля;
- запуск модуля;
- детальный разбор прошлых запусков

на одной странице.

### Как открывается история запусков

На странице редактирования DB-модуля должна быть отдельная кнопка:

- `История запусков`

Эта кнопка открывает отдельную страницу:

- `История запусков: <moduleTitle>`

На странице истории должна быть кнопка возврата:

- `К модулю`

### Что это за страница

Это страница истории конкретного DB-модуля, а не глобальная история по всем модулям.

Причина:

- основной сценарий пользователя — разбирать запуски конкретного модуля;
- не нужно сразу проектировать глобальный аналитический экран;
- меньше сложность маршрутизации и фильтрации.

### Структура страницы

Минимальный layout:

1. заголовок страницы:
   - `История запусков: <moduleTitle>`
2. кнопка `К модулю`
3. краткая шапка модуля:
   - `moduleCode`
   - `moduleTitle`
   - текущая ревизия
   - есть ли активный запуск
4. блок фильтров истории
5. список запусков
6. детальная панель выбранного запуска

### Фильтры истории

На первом этапе считаем достаточными фильтры:

- `Все`
- `Активные`
- `Успешные`
- `С ошибкой`

Дополнительно:

- строка поиска не нужна на первом этапе;
- фильтр по пользователю не нужен на первом этапе;
- фильтр по `launchSourceKind` можно отложить.

### Как должен выглядеть список запусков

Каждый элемент списка должен показывать:

- статус запуска;
- время запроса запуска;
- кто запустил;
- `launchSourceKind`:
  - `CURRENT_REVISION`
  - `WORKING_COPY`
- краткую summary-строку:
  - merge mode
  - строк в merged
  - target status

Если запуск активный:

- элемент списка должен быть явно помечен как `RUNNING`;
- активный запуск должен визуально выделяться.

### Как выбирать запуск по умолчанию

При открытии страницы:

- если у модуля есть активный запуск, по умолчанию выбираем самый свежий активный запуск;
- если активного запуска нет, выбираем самый свежий завершенный запуск;
- если запусков нет, показываем empty-state.

### Empty-state

Если запусков у DB-модуля нет, страница должна показывать понятное пустое состояние:

- `Для этого модуля еще нет запусков в режиме DB.`

И больше ничего не симулировать.

### Детальная панель выбранного запуска

В правой части страницы должен показываться один выбранный запуск.

Минимальные блоки:

1. `Итог запуска`
2. `Состояние источников`
3. `Ход выполнения`
4. `Артефакты`
5. `Execution snapshot` как технический блок

### Блок `Итог запуска`

Это компактная summary-таблица, без крупных metric-cards.

Что должно быть в таблице:

- статус запуска
- кто запустил
- когда запустил
- когда стартовал
- когда завершился
- `launchSourceKind`
- `mergeMode`
- `mergedRowCount`
- `successfulSourceCount`
- `failedSourceCount`
- `targetEnabled`
- `targetStatus`
- `targetTableName`
- `targetRowsLoaded`
- `outputDir`

Правило отображения:

- успешные статусы выделяются зеленым;
- ошибочные — красным;
- важные поля вроде target status и source counts — визуально акцентируются.

### Блок `Состояние источников`

Должен показывать source-level результаты выбранного запуска:

- source name
- status
- exported rows
- merged rows
- error message, если есть

Это должен быть компактный список/таблица без лишнего пустого пространства.

### Блок `Ход выполнения`

Должен строиться по `module_run_event`.

Правила:

- события показываются в порядке `seq_no`;
- успешные завершающие события подсвечиваются зеленым;
- ошибочные события подсвечиваются красным;
- нейтральные события остаются обычными.

Это должно быть прямое развитие текущего UX file-history, но уже поверх DB-данных.

### Блок `Артефакты`

Должен показывать:

- artifact kind
- artifact key
- file path
- storage status
- file size

Если файл уже удален:

- это должно быть явно видно по `storage_status`;
- история запуска при этом остается доступной.

### Блок `Execution snapshot`

Это технический блок.

По умолчанию он должен быть свернут.

Назначение:

- показать, по чему именно был выполнен запуск;
- дать доступ к `executionSnapshotId`;
- показать:
  - `sourceRevisionId`
  - `sourceWorkingCopyId`
  - `contentHash`
  - при необходимости raw YAML/JSON snapshot.

Этот блок не нужен большинству пользователей постоянно, но он важен для диагностики.

### Как обновлять активный запуск

Если выбранный запуск имеет статус `RUNNING`:

- страница должна периодически обновлять:
  - summary selected run
  - source results
  - timeline
  - artifacts

На первом этапе достаточно polling.

Рекомендация:

- polling раз в 3-5 секунд;
- после перехода запуска в терминальный статус polling останавливается.

### Что считать важным в UX этой страницы

Страница должна быть ориентирована не на редактирование, а на разбор факта запуска.

Поэтому:

- никаких полей редактирования на ней быть не должно;
- `run history` и `module editor` не смешиваем;
- если пользователь хочет запустить модуль снова, он возвращается на страницу модуля.

### Что считаем неправильным

Не хотим:

- открывать историю запусков в модальном окне;
- смешивать run history со страницей редактирования;
- рендерить только raw JSON без структурированного summary;
- автоматически скрывать факт, что запуск шел по `WORKING_COPY`;
- делать timeline только из текстовых логов без `module_run_event`.

## UX cleanup истории запусков для режима `database`

Cleanup истории запусков должен быть общей служебной операцией, а не действием внутри истории конкретного модуля.

Поэтому на текущем этапе считаем правильным размещать cleanup на странице:

- `Импорт моделей в БД`

как отдельный сервисный блок:

- `Очистка истории запусков`

### Что должно быть в блоке cleanup

Минимальный состав:

1. поле даты/времени:
   - `Удалить все запуски старше`
2. чекбокс:
   - `Сохранять минимум 30 запусков на каждый модуль`
   - включен по умолчанию
3. кнопка:
   - `Очистить историю запусков`

Дополнительно:

- рядом с полем даты должно быть видно, что по умолчанию используется один месяц назад;
- рядом с чекбоксом должна быть короткая подсказка, что при включенном флаге модули с `30` запусками и меньше не будут очищаться.

### Как должно работать значение по умолчанию

Если пользователь не меняет дату вручную:

- UI подставляет default:
  - текущий момент минус один календарный месяц.

Если пользователь не меняет флаг сохранения минимума:

- UI подставляет:
  - `preserveMinRunsPerModule = true`
  - `minRunsPerModule = 30`

### Подтверждение cleanup

Cleanup должен требовать confirm-диалог.

В confirm нужно явно показывать:

- что операция выполняется по всем DB-модулям сразу;
- выбранную cutoff date;
- включен ли safeguard по минимуму запусков;
- предупреждение о том, что файлы артефактов на диске тоже будут удаляться.

### Поведение при включенном safeguard

Если включен чекбокс `Сохранять минимум 30 запусков на каждый модуль`:

- cleanup не должен удалять историю модулей, у которых запусков `30` или меньше;
- для модулей с большим числом запусков cleanup не должен уменьшать историю ниже `30` последних запусков.

UI должен объяснять это простым текстом:

- `По умолчанию операция сохраняет минимум 30 запусков на каждый модуль.`

### Поведение при отключенном safeguard

Если пользователь снимает чекбокс:

- cleanup работает только по дате;
- история модуля может быть очищена полностью, если все его запуски старше cutoff date.

В этом случае confirm должен быть более жестким.

Текст подтверждения должен явно предупреждать:

- `Ограничение на минимум запусков отключено. История отдельных модулей может быть удалена полностью.`

### Что показывать после завершения cleanup

После успешной операции нужен отдельный result block.

Минимально:

- `cutoffDateTime`
- `preserveMinRunsPerModule`
- `minRunsPerModule`
- `deletedRunCount`
- `deletedExecutionSnapshotCount`
- `deletedArtifactFileCount`
- `failedArtifactFileDeleteCount`
- `skippedRunningRunCount`
- `protectedModuleCount`
- `protectedRunCount`
- список ошибок, если они были

### Что считать важным в UX cleanup

Пользователь должен понимать три вещи:

1. cleanup глобальный, а не по одному модулю;
2. по умолчанию есть защита от "слишком агрессивной" очистки;
3. при отключении этой защиты можно очистить историю модуля полностью.

### Что считаем неправильным

Не хотим:

- размещать cleanup внутри страницы истории конкретного модуля;
- запускать cleanup без confirm;
- скрывать, что очистка удаляет и файлы артефактов;
- по умолчанию очищать модули ниже порога в `30` запусков;
- делать safeguard обязательным без возможности его отключить.

## Что считаем неправильным

В черновике отдельно фиксируем, что не хотим:

- список модулей из БД, а детали из файлов;
- список из файлов, а запуск из БД;
- множественные `if (mode == files) else ...` по всему UI-коду;
- прямую зависимость запуска модуля от файловой структуры в режиме `database`.

Идея:

- выбран один режим;
- все module operations идут только через соответствующий store этого режима.

## Ограничения, зафиксированные на сейчас

- PostgreSQL для режима `database` обязателен;
- режимов только два: `files` и `database`;
- без гибридного режима;
- пока не удаляем `apps`;
- права доступа и разграничение ролей в режиме `database` на этом этапе не прорабатываем;
- считаем, что пользователь с доступом к UI и PostgreSQL может выполнять все допустимые операции режима `database`;
- сначала прорабатываем детали и только потом вносим это в `ROADMAP.md` и `BACKLOG.md`.
