# SQL Console Failure Scenarios

Цель документа:

- зафиксировать аварийные сценарии SQL-консоли;
- отдельно описать текущее поведение;
- определить безопасное целевое поведение;
- сформулировать инварианты: не вредить БД, не оставлять висящие транзакции, не держать скрытые блокировки.

## Ключевой принцип

Безопасность БД важнее удобства UI.

Если система не может надежно понять, жив ли пользовательский сеанс и подтвердит ли пользователь `Commit`, система обязана выбрать безопасный путь:

- не оставлять открытую транзакцию;
- не ждать бесконечно;
- откатить изменения автоматически, если транзакция осталась без владельца.

## Что происходит сейчас

Текущее поведение по коду:

- SQL-сценарий выполняется на стороне `ui-server`, а не в браузере;
- закрытие вкладки не останавливает серверный `async execution`;
- при `Autocommit = on` это обычно не оставляет висящую транзакцию;
- при `Autocommit = off` mutating-сценарий может дойти до состояния `PENDING_COMMIT`;
- в этом состоянии открытая транзакция держится в памяти `ui-server` и ждет явный `Commit` или `Rollback`.

Критичный вывод:

- если пользователь закрыл вкладку браузера, а сервер остался жив, `PENDING_COMMIT` сейчас может оставить открытую транзакцию и блокировки в БД;
- это неприемлемо и должно быть устранено отдельной доработкой.

## Инварианты безопасности

Ниже правила, которые нельзя нарушать:

- закрытие вкладки, refresh страницы или потеря сети не должны оставлять открытую транзакцию;
- браузер не должен быть единственным источником истины о том, можно ли держать транзакцию открытой;
- `PENDING_COMMIT` допустим только пока у transaction session есть живой владелец;
- у любой ручной транзакции должен быть ограниченный срок жизни;
- auto-rollback должен быть штатным и ожидаемым механизмом защиты, а не исключением;
- `Commit` и `Rollback` должны быть доступны только владельцу текущей execution session;
- read-only сценарии не должны попадать в `PENDING_COMMIT`.

## Аварийные сценарии

### 1. Пользователь закрыл вкладку во время `RUNNING`

Текущее поведение:

- серверный сценарий продолжает выполняться;
- если это manual-transaction сценарий с изменением данных, после успешного завершения он может перейти в `PENDING_COMMIT` без живого UI.

Безопасное поведение:

- закрытие вкладки должно означать потерю владельца execution session;
- если сценарий работал в ручной транзакции и владелец исчез, система не должна оставлять транзакцию в подвешенном состоянии;
- после завершения такого сценария должен происходить `auto-rollback`, а не переход в долгоживущий `PENDING_COMMIT`.

### 2. Пользователь закрыл вкладку во время `PENDING_COMMIT`

Текущее поведение:

- открытая транзакция может продолжать жить, пока жив процесс `ui-server`.

Безопасное поведение:

- при потере владельца должен запускаться короткий lease-timeout;
- после истечения lease система обязана сделать `auto-rollback`;
- результат должен явно помечаться как `ROLLED_BACK_BY_TIMEOUT` или аналогичным системным статусом.

### 3. Пользователь обновил страницу

Текущее поведение:

- серверный execution не теряется;
- при ручной транзакции возможна гонка между старой вкладкой, новой вкладкой и pending session.

Безопасное поведение:

- refresh должен либо восстанавливать владение той же session, либо приводить к выпуску нового owner token;
- старая session не должна жить без heartbeat бесконечно.

### 4. Потеря сети между браузером и `ui-server`

Текущее поведение:

- сервер не знает, исчез пользователь временно или окончательно;
- manual transaction может зависнуть.

Безопасное поведение:

- нужен heartbeat от UI к серверу;
- heartbeat должен обновлять lease владельца;
- если heartbeat не пришел вовремя, pending transaction должна быть автоматически откатена.

### 5. Падение или рестарт `ui-server`

Текущее поведение:

- PostgreSQL обычно откатывает незавершенную транзакцию при закрытии соединения;
- но система сейчас не документирует это как гарантированное поведение верхнего уровня.

Безопасное поведение:

- рестарт сервера не должен приводить к коммиту;
- транзакция должна гарантированно завершаться rollback на уровне закрытия JDBC connection;
- на уровне продукта это нужно считать защитным fallback, но не основной бизнес-логикой.

### 6. Второй клиент, вторая вкладка или другой пользователь

Текущее поведение:

- нужно считать это небезопасной областью, пока у execution session нет явного owner token.

Безопасное поведение:

- `Commit` и `Rollback` должны требовать подтвержденный owner token;
- один execution session должен иметь одного владельца;
- если нужен handover между вкладками, это должно быть отдельным, явным механизмом.

### 7. Ошибка БД или разрыв соединения во время script execution

Текущее поведение:

- сценарий завершается ошибкой;
- для ручной транзакции важно убедиться, что соединение не оставляет открытое состояние.

Безопасное поведение:

- при любой ошибке transport/JDBC/driver/connection loss ручная транзакция должна считаться сорванной и завершаться rollback;
- после этого session не должна оставаться в `PENDING_COMMIT`.

## Целевое проектное решение

### Обязательно

- server-side owner session для каждого async execution;
- owner token для `Commit` / `Rollback`;
- heartbeat из UI во время `RUNNING` и `PENDING_COMMIT`;
- lease timeout на владельца;
- hard TTL на `PENDING_COMMIT`, даже если heartbeat продолжает приходить;
- automatic rollback при потере владельца или превышении TTL;
- audit/event log с причиной системного rollback.

### Допустимо как best-effort, но не как основная защита

- `beforeunload` / `sendBeacon` из браузера;
- попытка мягко отменить execution при закрытии вкладки.

### Недопустимо

- бесконечно ждать `Commit` от уже закрытой вкладки;
- держать transaction session только на основании того, что процесс сервера еще жив;
- разрешать `Commit`/`Rollback` без подтверждения владельца session.

### Terminal control-path invariant

После terminal state SQL execution больше не имеет active browser owner session.

Terminal states включают:

- обычное завершение async execution;
- explicit `Commit`;
- explicit `Rollback`;
- system rollback по timeout или потере владельца;
- cancel/failure, если execution уже перешел в финальное состояние.

Ожидаемое поведение:

- public terminal snapshot не должен возвращать active control-path metadata:
  - `ownerToken`;
  - `ownerLeaseExpiresAt`;
  - `pendingCommitExpiresAt`;
- stale browser tab не должен получать успешный ответ на terminal control action;
- `/heartbeat` после terminal state должен возвращать conflict, потому что heartbeat больше не нужен;
- `/cancel` после terminal state должен возвращать conflict, потому что execution уже завершен;
- `/release` после terminal state должен возвращать conflict, потому что active control-path session уже отсутствует;
- повторные `/commit` и `/rollback` после final manual transaction должны возвращать conflict, потому что незавершенной транзакции уже нет.

Это не отменяет owner-token model для active states. В `RUNNING` и `PENDING_COMMIT` owner token, heartbeat и explicit actions остаются обязательной частью safety-механизма.

## Минимальный безопасный план реализации

### Этап 1

- добавить owner session и owner token;
- добавить heartbeat;
- добавить lease timeout;
- при истечении lease делать auto-rollback для `PENDING_COMMIT`.

### Этап 2

- если manual-transaction сценарий завершился успешно, но owner уже потерян, не переводить его в долгий `PENDING_COMMIT`, а откатывать автоматически;
- добавить явные пользовательские статусы системного rollback.

### Этап 3

- закрепить поведение тестами на:
  - tab close;
  - refresh;
  - network loss;
  - duplicate tab;
  - server restart.

## Решение по умолчанию

Если система сомневается, жив ли пользователь и можно ли ждать подтверждения, она должна делать `Rollback`.
