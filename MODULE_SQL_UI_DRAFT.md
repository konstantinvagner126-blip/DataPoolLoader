# Unified Module Editor Draft

Черновик. Не является утвержденной частью roadmap/backlog.

## 1. Опорная модель

Для UI модуль в обоих режимах является одной и той же сущностью:

- конфиг модуля;
- SQL-ресурсы модуля;
- UI-метаданные;
- история запусков;
- служебное состояние редактирования.

Различается только механизм хранения:

- `files`
- `database`

Короткая формула:

`Module = config + sql resources + metadata`

`FILES/DB = storage implementation`

Из этого следуют два базовых правила:

- не должно быть двух разных редакторов модуля;
- должен быть один `Module Editor` с разными storage adapters.

## 2. Что считаем уже выбранным

Эти решения в рамках черновика считаем принятыми:

- `FILES` и `DB` используют один и тот же визуальный редактор;
- различаются только storage/lifecycle-операции;
- визуальная форма не должна зеркалить YAML один в один;
- raw `application.yml` остается в UI как технический fallback;
- вкладка `SQL` является каталогом SQL-ресурсов модуля;
- создание, переименование и удаление SQL делаются на вкладке `SQL`;
- во вкладке `Настройки` пользователь выбирает источник SQL, а не редактирует поля `sql/sqlFile` напрямую;
- YAML-схема не меняется, меняется только пользовательская модель редактирования.

## 3. Цель дизайна

Сделать единый редактор модуля, который:

- одинаково выглядит в `files` и `database`;
- понятен человеку, а не только YAML-парсеру;
- позволяет штатно работать с конфигом и SQL без знания storage-деталей;
- не ломает существующую конфигурационную модель;
- оставляет технический выход в `application.yml` для нестандартных случаев.

## 4. Целевая структура редактора

Редактор модуля должен иметь единый shell.

### 4.1 Верхняя зона

Верхняя зона всегда содержит:

- название модуля;
- краткое описание;
- индикатор режима хранения;
- статус изменений;
- action bar.

Общие действия:

- `Запустить`
- `Отменить изменения`

Для `files`:

- `Сохранить`

Для `database`:

- `Сохранить рабочую копию`
- `Удалить рабочую копию`
- `Опубликовать`

Принцип:

- layout одинаковый;
- отличаются только lifecycle-кнопки.

### 4.2 Вкладки редактора

Фиксируем такой состав вкладок:

- `Настройки`
- `SQL`
- `application.yml`
- `Метаданные`

История запусков, summary, диагностика и lifecycle-информация не являются вкладками редактора. Они остаются отдельными панелями shell.

## 5. Оптимальная форма редактирования

Ключевой принцип:

- пользователь редактирует не поля `commonSql/commonSqlFile/sql/sqlFile`;
- пользователь редактирует источник SQL для default и для каждого source.

### 5.1 Структура вкладки `Настройки`

Во вкладке `Настройки` должны быть такие секции:

- `Общие параметры`
- `SQL по умолчанию`
- `Источники`
- `Квоты`
- `Target`

Это должна быть одна и та же форма для `files` и `database`.

### 5.2 Секция `Общие параметры`

Секция содержит обычные поля исполнения:

- `outputDir`
- `fileFormat`
- `mergeMode`
- `errorMode`
- `parallelism`
- `fetchSize`
- `queryTimeoutSec`
- `progressLogEveryRows`
- `maxMergedRows`
- `deleteOutputFilesAfterCompletion`

Это остается почти как сейчас. Основное изменение требуется не здесь, а в SQL-модели формы.

### 5.3 Секция `SQL по умолчанию`

Вместо raw-полей `commonSql` и `commonSqlFile` показываем один логический блок.

Блок должен содержать:

- selector `Источник SQL`
- краткое пояснение effective behavior
- редактор/selector в зависимости от выбранного режима

Фиксируем три режима:

- `Не задан`
- `Inline SQL`
- `SQL из каталога`

Поведение:

`Не задан`

- default SQL отсутствует;
- `commonSql = ""`
- `commonSqlFile = null`

`Inline SQL`

- показывается компактный SQL editor;
- `commonSql = <text>`
- `commonSqlFile = null`

`SQL из каталога`

- показывается select SQL-ресурса из вкладки `SQL`;
- `commonSql = ""`
- `commonSqlFile = <sqlResourceKey>`

Дополнительно в этом блоке всегда показывается summary:

- `Используется для источников без собственного SQL`

### 5.4 Секция `Источники`

Каждый source должен отображаться карточкой.

Карточка source содержит:

- имя источника;
- JDBC-параметры;
- блок `SQL источника`;
- краткую summary-строку с effective SQL behavior.

Для `SQL источника` фиксируем три штатных режима:

- `Наследовать SQL по умолчанию`
- `Inline SQL`
- `SQL из каталога`

Поведение:

`Наследовать SQL по умолчанию`

- `source.sql = null`
- `source.sqlFile = null`

`Inline SQL`

- показывается компактный SQL editor;
- `source.sql = <text>`
- `source.sqlFile = null`

`SQL из каталога`

- показывается select SQL-ресурса;
- `source.sql = null`
- `source.sqlFile = <sqlResourceKey>`

Summary-строка должна показывать один из статусов:

- `Наследует SQL по умолчанию`
- `Использует inline SQL`
- `Использует SQL-ресурс: <name>`
- `Использует внешнюю SQL-ссылку`

### 5.5 Как должен выглядеть inline SQL

Inline SQL нужен, но не должен раздувать форму.

Фиксируем такой принцип:

- inline SQL в форме показывается только когда реально выбран режим `Inline SQL`;
- это компактный редактор на ограниченную высоту;
- если SQL большой, пользователь может перейти в `application.yml`, но не обязан это делать;
- форма не должна постоянно занимать пол-экрана пустыми SQL-редакторами.

## 6. Вкладка `SQL`

Вкладка `SQL` является каталогом SQL-ресурсов модуля.

Это не “просмотр файлов”, а рабочий реестр SQL, который связан с формой.

### 6.1 Layout

Фиксируем layout:

- слева список SQL-ресурсов;
- справа редактор выбранного SQL;
- над редактором краткая информация о ресурсе и его использовании.

### 6.2 Что видно в списке SQL-ресурсов

Для каждого SQL-ресурса показываем:

- имя ресурса;
- secondary descriptor:
  - для `files` это путь;
  - для `database` это key asset'а;
- usage summary:
  - `SQL по умолчанию`
  - `Источник: <name>`
  - `Используется в нескольких местах`
  - `Не используется`

Главное правило:

- primary label не должен дублироваться как `label + path`, если это одно и то же значение.

### 6.3 Действия на вкладке `SQL`

Фиксируем:

- `Создать SQL`
- `Переименовать`
- `Удалить`

Принцип CRUD:

- новые SQL-ресурсы создаются только здесь;
- форма `Настройки` ничего не создает, а только выбирает;
- rename автоматически обновляет ссылки:
  - `commonSqlFile`
  - `source.sqlFile`
- удалить можно только неиспользуемый ресурс;
- если ресурс используется, UI показывает места использования и не дает удалить его до отвязки.

### 6.4 Что считается допустимым

Допустимо иметь SQL-ресурсы со статусом:

- `Не используется`

Это не ошибка. Пользователь может готовить SQL заранее.

## 7. Вкладка `application.yml`

Эта вкладка обязательна как технический fallback.

Назначение:

- работа с редкими или нестандартными случаями;
- ручная правка YAML;
- восстановление после конфликтной или частично невалидной конфигурации.

Принцип:

- это не primary UI;
- но без нее редактор будет хрупким и неполным.

Все нестандартные кейсы должны быть видны в visual form через warning, но исправляться при необходимости можно через `application.yml`.

## 8. Вкладка `Метаданные`

Отдельная общая вкладка для:

- `title`
- `description`
- `tags`
- `hiddenFromUi`

Она не должна зависеть от режима хранения.

## 9. Нештатные и пограничные случаи

### 9.1 Внешние SQL-ссылки в `files`

Если YAML содержит `sqlFile`, который не относится к SQL-ресурсам модуля:

- visual form не должна пытаться полноценно управлять такой ссылкой;
- в форме показываем статус:
  - `Внешняя SQL-ссылка`
- рядом даем понятную подсказку:
  - `Редактируется только через application.yml`

То есть visual editor не ломает конфиг, но и не делает вид, что умеет безопасно его сопровождать.

### 9.2 Внешние SQL-ссылки в `database`

Для `database` это считаем нештатной конфигурацией.

Правило:

- visual editor не предлагает такие ссылки;
- если они уже есть в raw YAML, форма показывает warning;
- normal visual flow для DB должен работать только с:
  - inline SQL;
  - SQL-ресурсами самого модуля.

### 9.3 Отсутствующий SQL-ресурс

Если конфиг ссылается на SQL-ресурс, которого нет:

- форма должна явно показать проблему;
- selector остается в проблемном состоянии;
- warning должен быть виден без открытия `application.yml`.

## 10. Внутренняя UI-модель

Для формы нужен не прямой raw-state из YAML, а промежуточный visual state.

Минимальный набор полей:

- `defaultSqlMode`
- `defaultSqlInlineText`
- `defaultSqlResourceKey`
- для каждого source:
  - `sqlMode`
  - `sqlInlineText`
  - `sqlResourceKey`
  - `sqlExternalReference`

Именно этот state должен:

- строиться из YAML;
- редактироваться формой;
- сериализоваться обратно в YAML.

## 11. Что должно быть общим в коде

На уровне архитектуры надо выделить общие контракты:

- `ModuleEditor`
- `ModuleConfigForm`
- `SqlResourceCatalog`
- `SqlResourceUsage`

Storage adapter должен отвечать только за:

- чтение модуля;
- сохранение;
- список SQL-ресурсов;
- create/rename/delete SQL;
- lifecycle-операции режима.

Форма, layout и правила выбора SQL не должны дублироваться между `files` и `database`.

## 12. Backend-контракты

Для backend нужно зафиксировать такое же правило, как для UI:

- если механизм существует в обоих режимах, он должен быть выражен общим интерфейсом;
- у этого интерфейса должно быть две реализации:
  - `files`
  - `database`
- DB-only механизмы не нужно искусственно оборачивать в file-реализацию ради симметрии.

То есть цель не “интерфейс на каждый класс”, а общий contract на каждую общую capability редактора.

### 12.1 Что уже частично сделано

У нас уже есть правильное направление в части запуска:

- `ModuleExecutionSource`
- порты ядра вокруг `ApplicationRunner`

Это хороший паттерн и его нужно довести до storage/editor слоя.

### 12.2 Какие backend-механизмы обязаны иметь общий контракт

Фиксируем минимальный набор общих backend-capabilities.

#### `ModuleCatalogService`

Отвечает за:

- список модулей;
- получение карточки модуля;
- загрузку деталей модуля;
- создание модуля;
- удаление модуля.

Должно быть две реализации:

- `FilesModuleCatalogService`
- `DatabaseModuleCatalogService`

#### `ModuleEditorStore`

Отвечает за содержимое редактируемого модуля:

- загрузку `config + sql resources + metadata`;
- сохранение изменений;
- валидацию существования связанных SQL-ресурсов;
- преобразование storage-data в единый editor model.

Должно быть две реализации:

- `FilesModuleEditorStore`
- `DatabaseModuleEditorStore`

Это основной backend-контракт для общего `Module Editor`.

#### `SqlResourceService`

Отвечает за каталог SQL-ресурсов:

- список ресурсов;
- чтение содержимого SQL;
- создание SQL;
- переименование SQL;
- удаление SQL;
- расчет usage.

Должно быть две реализации:

- `FilesSqlResourceService`
- `DatabaseSqlResourceService`

#### `ModuleLifecycleService`

Отвечает за mode-specific lifecycle действия.

Общие операции задаются контрактом через capability model:

- `save`
- `discardChanges`
- `publish`
- `deleteWorkingCopy`

Важно:

- не каждая операция должна быть доступна в каждой реализации;
- интерфейс должен уметь отдавать capabilities текущего storage mode;
- UI не должен угадывать, какие кнопки показывать.

Должно быть две реализации:

- `FilesModuleLifecycleService`
- `DatabaseModuleLifecycleService`

Поведение:

- в `files` часть операций будет unsupported по capability;
- в `database` будут доступны `working copy / publish`.

#### `ModuleRunService`

Отвечает за запуск модуля из UI.

Контракт должен скрывать различие между:

- запуском файлового модуля;
- запуском DB-модуля.

Внутри он использует общий `ApplicationRunner`, но с разными storage adapters.

Должно быть две реализации:

- `FilesModuleRunService`
- `DatabaseModuleRunService`

#### `ModuleRunHistoryService`

Отвечает за выдачу истории запусков и деталей запуска для конкретного режима хранения.

Должно быть две реализации:

- `FilesModuleRunHistoryService`
- `DatabaseModuleRunHistoryService`

Даже если хранение истории устроено по-разному, UI должен получать один и тот же contract.

### 12.3 Что не нужно насильно делать общим

Не нужно искусственно обобщать механизмы, которые реально существуют только в `database`.

Примеры:

- импорт модулей `files -> db`;
- sync в БД;
- maintenance lock на массовый импорт;
- специфичные DB migration/mechanisms.

Для таких сценариев нормально иметь только DB-реализацию.

### 12.4 Главный backend-принцип

Правильная архитектура здесь такая:

- один общий editor/use-case layer;
- общий набор backend contracts;
- две storage-реализации для общих механизмов;
- DB-only процессы остаются отдельными сервисами.

То есть:

- UI не должен напрямую знать про `ModuleRegistry` и `DatabaseModuleStore` как про два разных мира;
- UI/use-case слой должен работать через общий contract;
- выбор конкретной реализации должен происходить на уровне composition root.

## 13. Итоговая форма, которую считаем оптимальной

Оптимальной считаем такую форму:

- один общий редактор модуля для `FILES/DB`;
- вкладка `Настройки` работает через модель `Источник SQL`, а не через raw YAML-поля;
- вкладка `SQL` является отдельным каталогом SQL-ресурсов;
- `application.yml` остается технической вкладкой;
- различие между `FILES` и `DB` скрыто внутри storage adapter;
- пользователь в обоих режимах решает одну и ту же задачу:
  - редактирует конфиг;
  - управляет SQL-ресурсами;
  - видит, какой SQL реально используется.

## 14. Что потом пойдет в backlog

Когда этот draft будет принят, из него нужно будет сделать отдельный блок задач:

- общий `Module Editor` contract;
- visual SQL state и serializer;
- refactoring формы `Настройки`;
- refactoring вкладки `SQL` в каталог ресурсов;
- warnings для external/missing SQL refs;
- file/db adapters для SQL resources;
- backend contracts для editor/storage/run/history;
- выравнивание UX `FILES/DB` на общем editor shell.
