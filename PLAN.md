# DataPoolLoader Plan

## Цель

Разработать Kotlin CLI-приложение, которое:

- читает из YAML-конфига список PostgreSQL БД произвольного размера;
- выполняет на каждой БД указанный `SELECT`;
- сохраняет результат каждой БД в отдельный файл;
- формирует общий результирующий файл;
- загружает итоговый результирующий файл в отдельную PostgreSQL БД в указанную таблицу;
- поддерживает режим равномерного чередования строк из разных БД (`round_robin`);
- корректно работает на больших объемах данных без полной загрузки результата в память.

## Ключевые функциональные требования

### 1. Конфигурация

В `application.yml` должны задаваться:

- список подключений к PostgreSQL;
- общий SQL-запрос для выгрузки;
- опциональное переопределение SQL на уровне конкретного источника;
- подключение к целевой PostgreSQL БД для загрузки результата;
- имя таблицы в целевой БД;
- флаг, нужно ли очищать целевую таблицу перед загрузкой;
- флаг, нужно ли вообще выполнять загрузку в целевую БД;
- директория для выходных файлов;
- формат выходных файлов;
- режим объединения;
- параметры отказоустойчивости;
- параметры параллелизма;
- параметры JDBC-fetch/streaming.

### 2. Подключение к БД

Приложение должно:

- подключаться к каждой БД из конфига;
- выполнять только `SELECT`;
- валидировать, что в конфиге не передан небезопасный SQL для модификации данных;
- выполнять выгрузки параллельно с ограничением по числу одновременных задач;
- логировать старт, завершение и ошибки по каждой БД.

### 3. Выгрузка по каждой БД

Для каждой БД нужно:

- выполнить запрос потоково;
- сохранить результат в отдельный файл;
- сохранить метаданные выгрузки: имя источника, время запуска, число выгруженных строк, статус.

Если источник вернул 0 строк:

- это считается успешной выгрузкой;
- файл создается с заголовком CSV;
- в лог выводится информация о нулевом результате.

Если одна из БД завершилась ошибкой:

- ошибка фиксируется в логах и summary;
- остальные источники продолжают обрабатываться;
- итоговый файл формируется из всех успешных выгрузок.

### 4. Общая выгрузка

Нужно поддержать два режима:

- `plain`:
  общий файл формируется последовательной записью результатов источников;
- `round_robin`:
  общий файл формируется по одной строке из каждого источника по кругу, пока строки не закончатся.

Пример `round_robin`:

- DB1: `A1 A2 A3`
- DB2: `B1 B2`
- DB3: `C1 C2 C3 C4`

Результат:

`A1 B1 C1 A2 B2 C2 A3 C3 C4`

### 5. Загрузка результата в целевую БД

После формирования `merged.csv` приложение должно:

- подключиться к целевой PostgreSQL БД из конфига;
- выполнить preflight-проверку совместимости target таблицы и incoming колонок;
- при включенном флаге выполнить `TRUNCATE TABLE`;
- загрузить данные из `merged.csv` в указанную таблицу;
- логировать старт и завершение загрузки;
- фиксировать ошибку загрузки как ошибку запуска.

Если загрузка отключена флагом:

- `merged.csv` и `summary.json` все равно формируются;
- импорт в target БД не выполняется.

## Нефункциональные требования

### 1. Работа с большими объемами

Ожидаются выборки от 100 000+ строк на одну БД, поэтому:

- нельзя собирать все строки всех БД в память;
- чтение из PostgreSQL должно быть потоковым через JDBC `ResultSet`;
- запись в отдельные файлы должна идти построчно;
- сборка общего файла должна идти потоково.

### 2. Стратегия по памяти

Принцип реализации:

- сначала каждая БД выгружается в отдельный файл;
- затем общий файл строится на основе уже созданных файлов;
- для `plain` читаем источники по очереди;
- для `round_robin` открываем все промежуточные файлы и читаем по одной строке из каждого файла по кругу.
- загрузку в целевую БД выполняем напрямую из `merged.csv`, без повторного чтения всего файла в память.

Это позволяет держать в памяти только текущую строку или небольшой буфер, а не полный результат.

### 3. Отказоустойчивость

Базовая модель обработки ошибок:

- при ошибке одной БД процесс не останавливается;
- ошибка фиксируется в логах и summary;
- итоговый merge выполняется по успешным источникам;
- если успешных источников нет, процесс завершается ошибкой.

### 4. Параллельность

Выгрузка по БД должна выполняться параллельно.

Для устойчивости и контроля нагрузки предлагается:

- использовать ограниченный пул потоков;
- настраивать максимальный параллелизм через конфиг;
- не запускать неограниченное число JDBC-подключений одновременно.

## Предлагаемый стек

Базовый вариант без Spring:

- Kotlin JVM;
- Gradle;
- PostgreSQL JDBC Driver;
- PostgreSQL `COPY FROM STDIN` для быстрой загрузки итогового CSV;
- Jackson YAML или SnakeYAML для конфига;
- SLF4J + Logback для логирования;
- JUnit 5 для тестов.

### Почему без Spring

Для первой версии это batch/CLI-утилита, а не сервис:

- не нужен HTTP API;
- не нужен embedded server;
- не нужна сложная автоконфигурация;
- лучше сохранить минимальный runtime и простую структуру.

Если в будущем появятся API, scheduler, метрики или сервисный deployment, можно будет рассмотреть миграцию на Spring Boot.

## Формат данных

Для первой версии предлагаю использовать `CSV`.

Плюсы:

- удобно анализировать вручную;
- легко открывать в табличных инструментах;
- удобно объединять построчно;
- меньше накладных расходов, чем у JSON.

Формат первой версии:

- кодировка `UTF-8`;
- разделитель `,`;
- заголовок колонок записывается в каждый отдельный файл и в общий файл.

Ограничение:

- все БД должны возвращать одинаковый состав колонок в одинаковом порядке.

Это требование нужно явно валидировать:

- сравнивать список колонок первой успешной выгрузки с остальными;
- при несовпадении помечать источник как несовместимый, логировать это и исключать его из merge.

## Предварительная структура `application.yml`

```yaml
app:
  outputDir: ./output
  fileFormat: csv
  mergeMode: round_robin
  errorMode: continue_on_error
  parallelism: 5
  fetchSize: 1000
  sql: |
    select id, created_at, payload
    from some_table
    where created_at >= current_date - interval '1 day'

  sources:
    - name: db1
      jdbcUrl: jdbc:postgresql://host1:5432/dbname
      username: user1
      password: ${DB1_PASSWORD}
      sql: |
        select id, created_at, payload
        from special_table

    - name: db2
      jdbcUrl: jdbc:postgresql://host2:5432/dbname
      username: user2
      password: local-dev-password

  target:
    enabled: true
    jdbcUrl: jdbc:postgresql://target-host:5432/target_db
    username: loader_user
    password: ${TARGET_DB_PASSWORD}
    table: public.test_data_pool
    truncateBeforeLoad: true
```

### Правила по секретам

- поля `jdbcUrl`, `username` и `password` могут содержать прямые значения;
- поля `jdbcUrl`, `username` и `password` могут содержать шаблон вида `${KEY}`;
- при использовании `${KEY}` приложение сначала читает значение из `credential.properties`, затем из переменных окружения, затем из JVM system properties;
- если значение для placeholder не найдено у источника, источник помечается ошибочным, ошибка логируется, обработка остальных БД продолжается;
- если значение для placeholder не найдено у целевой БД, запуск завершается ошибкой.

## Алгоритм работы приложения

### Этап 1. Загрузка конфигурации

- прочитать `application.yml`;
- провалидировать обязательные поля;
- провалидировать, что список `sources` не пустой;
- провалидировать общий `sql`;
- если у источника задан собственный `sql`, использовать его вместо общего;
- провалидировать обязательные поля `target`;
- провалидировать режимы `mergeMode` и `errorMode`.

### Этап 2. Выгрузка по источникам

Для каждого источника:

- открыть JDBC-соединение;
- включить streaming-friendly настройки;
- выполнить `SELECT`;
- считать метаданные колонок;
- записать заголовок CSV;
- построчно выгрузить строки в файл `<source>.csv`;
- посчитать число строк;
- сохранить результат выполнения в объекте summary.

Выполнение этого этапа должно идти параллельно:

- задачи выгрузки запускаются через ограниченный executor;
- степень параллелизма задается в конфиге;
- результат по каждому источнику собирается независимо.

### Этап 3. Проверка совместимости схем

После первых успешных выгрузок:

- сравнить имена и порядок колонок между файлами;
- если схемы не совпадают, обработать по выбранной политике ошибки.

### Этап 4. Формирование общего файла

#### Режим `plain`

- открыть итоговый файл;
- записать один общий заголовок;
- последовательно читать все успешные промежуточные файлы;
- копировать строки в итоговый файл.

#### Режим `round_robin`

- открыть итоговый файл;
- записать общий заголовок;
- открыть все успешные промежуточные файлы;
- пропустить заголовки;
- читать по одной строке из каждого файла по кругу;
- исключать файл из обхода, когда строки в нем закончились;
- продолжать, пока не исчерпаются все источники.

### Этап 5. Загрузка в целевую БД

- выполнить preflight-проверку колонок target таблицы;
- открыть соединение с целевой PostgreSQL БД;
- при `target.truncateBeforeLoad=true` выполнить `TRUNCATE TABLE`;
- сформировать команду `COPY <table> (<columns...>) FROM STDIN WITH (FORMAT csv, HEADER true)`;
- потоково передать в БД содержимое `merged.csv`;
- зафиксировать успешную загрузку в логах и `summary.json`.

Если `target.enabled=false`:

- этап импорта пропускается;
- это отражается в `summary.json`.

## Выходные артефакты

Для каждого запуска создавать директорию:

```text
output/<timestamp>/
```

Внутри:

- `<source-name>.csv` — отдельная выгрузка по БД;
- `merged.csv` — общий файл;
- `summary.json` — краткий отчет по запуску;
- логи выводятся в консоль.
