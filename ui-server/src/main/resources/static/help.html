Инструмент 1

Модуль загрузки данных

Используется для выгрузки данных из нескольких PostgreSQL-источников, сборки общего файла и загрузки результата в target.
Открыть модуль
Основной сценарий
Выгрузка → Объединение → Загрузка
Сначала данные выгружаются по каждому source, затем объединяются в merged.csv и при необходимости загружаются в target.
Где настраивается
application.yml
У каждого файлового модуля есть свой src/main/resources/application.yml и свой набор SQL-файлов.
Как запускать
UI / IDEA / Gradle
Модуль можно запускать прямо из UI или напрямую как обычный модуль проекта.
Файловый режим
Где хранятся модули: в каталоге apps.
Когда использовать: модуль редактируется как набор файлов и может запускаться напрямую из проекта.
Что важно: должен быть корректно настроен ui.appsRoot.
DB режим
Где хранятся модули: в базе данных.
Когда использовать: нужен личный черновик, публикация изменений, история запусков и совместная работа нескольких пользователей.
Что важно: сначала импортируй модуль из файлов или создай его прямо в UI.
Когда использовать
  • Нужно собрать общий пул данных из нескольких БД.
  • Нужно получить merged.csv для нагрузки.
  • Нужно загрузить результат в target-таблицу перед тестом.
  • Нужно регулярно запускать один и тот же сценарий по готовому конфигу.
Порядок действий
  1. Выбери модуль в UI.
  2. Проверь Настройки модуля или открой application.yml.
  3. При необходимости поправь SQL-файлы.
  4. Убедись, что найден credential.properties, если есть ${...}.
  5. Нажми Запустить и смотри историю, статусы и итог запуска.
Как работать в DB режиме
  1. Переключи UI в режим База данных на главной странице.
  2. Открой страницу модулей из базы данных и выбери модуль.
  3. Сохрани изменения в личный черновик.
  4. При необходимости опубликуй черновик как новую ревизию.
  5. Запускай модуль и смотри историю запусков, события, результаты по источникам и артефакты.
Как импортировать модуль из файлов в базу данных
  1. Перейди на страницу Импорт из файлов из DB-режима.
  2. Запусти массовый импорт всех модулей или импорт конкретного модуля.
  3. UI прочитает application.yml, SQL-ресурсы и UI-метаданные файлового модуля.
  4. Если код модуля уже есть в registry, импорт сравнит содержимое и либо обновит DB-модуль, либо пропустит его без изменений.
  5. После успешного импорта модуль появится на странице Модули из базы данных и будет готов к работе через личный черновик и публикацию.
Во время полного импорта DB-редактор и часть операций временно блокируются, чтобы не смешивать массовое обновление с ручными изменениями.
Администрирование DB-модуля
Новый модуль: создает пустой DB-модуль сразу в registry, без файлового каталога.
Сохранить черновик: сохраняет твои изменения как личную рабочую версию без публикации для других пользователей.
Опубликовать: делает личный черновик новой общей ревизией модуля.
Сбросить черновик: удаляет личную рабочую версию и возвращает редактор к опубликованной ревизии.
Удалить модуль: удаляет DB-модуль из registry, если он больше не нужен.
Скрыть из UI: модуль можно оставить в registry, но убрать из обычного каталога, если он служебный или подготовительный.
Что блокируется в DB режиме
Массовый импорт
Пока идет массовый импорт модулей, DB-редактор работает в ограниченном режиме: нельзя сохранять, публиковать и удалять модули.
Импорт одного модуля
Если импортируется один конкретный модуль, блокировка применяется только к нему. Остальные DB-модули остаются доступны.
Личный черновик
Публиковать или сбрасывать черновик можно только для выбранного модуля и только когда над ним не идет конфликтующая операция.
Запуски
История и результаты доступны всегда, но сам запуск модуля зависит от режима работы UI и от доступности PostgreSQL.
Какие разделы конфига важны
Общие параметры: outputDir, mergeMode, parallelism, fetchSize, queryTimeoutSec.
SQL: общий SQL через commonSql / commonSqlFile или отдельный SQL на уровне источника через sources[].sql / sources[].sqlFile.
Источники: sources[].name, jdbcUrl, username, password.
Квоты: quotas используются только для mergeMode=quota.
Целевая загрузка: enabled, jdbcUrl, username, password, table, truncateBeforeLoad.
Частые defaults
outputDir./output
mergeModeplain
parallelism5
fetchSize1000
progressLogEveryRows10000
target.enabledtrue
deleteOutputFilesAfterCompletionfalse
Минимальный пример application.yml
app:
  outputDir: ./output
  mergeMode: round_robin
  parallelism: 5
  fetchSize: 1000
  queryTimeoutSec: 60
  commonSqlFile: classpath:sql/common.sql

  sources:
    - name: db1
      jdbcUrl: ${DB1_JDBC_URL}
      username: ${DB1_USERNAME}
      password: ${DB1_PASSWORD}

    - name: db2
      jdbcUrl: ${DB2_JDBC_URL}
      username: ${DB2_USERNAME}
      password: ${DB2_PASSWORD}
      sqlFile: classpath:sql/db2_override.sql

  target:
    enabled: true
    jdbcUrl: ${TARGET_JDBC_URL}
    username: ${TARGET_USERNAME}
    password: ${TARGET_DB_PASSWORD}
    table: public.test_data_pool
    truncateBeforeLoad: true
Что чаще всего ломает запуск
  • credential.properties не найден, хотя в конфиге есть ${...}.
  • Одновременно заданы и inline SQL, и SQL-файл для одного уровня.
  • Неверно указан путь classpath:sql/....
  • Пустой или несуществующий target.table при включенном импорте.
  • Некорректный ui.appsRoot, если запуск идет через собранный UI.
Инструмент 2

SQL-консоль

Инструмент для ручной проверки данных и служебных операций по настроенным источникам без запуска полного batch-сценария.
Открыть SQL-консоль
Источники
ui.sqlConsole.sources
SQL-консоль работает не через файловые модули, а через свой список источников из UI-конфига.
Режим
Одна SQL-команда
Сейчас за один запуск выполняется одна SQL-команда. Для SELECT данные показываются по каждому источнику отдельно.
Управление
Фоновое выполнение и отмена
Запрос выполняется асинхронно, можно следить за статусом и отменять долгие операции.
Когда использовать
  • Нужно быстро проверить данные на конкретных источниках.
  • Нужно выполнить одиночный SELECT, UPDATE, CREATE или другую служебную SQL-команду.
  • Нужно сравнить ответы нескольких источников без полного запуска модуля.
  • Нужно вручную подготовить данные перед нагрузкой.
Порядок действий
  1. Открой SQL-консоль.
  2. Убедись, что настроены ui.sqlConsole.sources.
  3. Проверь credential.properties, если в источниках есть placeholders.
  4. Отметь нужные источники чекбоксами.
  5. Введи SQL и нажми Выполнить.
  6. При долгом выполнении используй кнопку Отменить.
Минимальный пример конфига
ui:
  sqlConsole:
    fetchSize: 1000
    maxRowsPerShard: 200
    queryTimeoutSec: 60
    sources:
      - name: db1
        jdbcUrl: ${DB1_JDBC_URL}
        username: ${DB1_USERNAME}
        password: ${DB1_PASSWORD}
      - name: db2
        jdbcUrl: ${DB2_JDBC_URL}
        username: ${DB2_USERNAME}
        password: ${DB2_PASSWORD}
Как выглядит результат
Для SELECT: отдельные результаты по каждому источнику, без смешивания строк между ними.
Для DDL / DML: статусы выполнения по каждому источнику, включая ошибки и affected rows, если они есть.
Ограничение: количество строк режется по maxRowsPerShard.
Timeout: используется ui.sqlConsole.queryTimeoutSec.
Общий компонент

credential.properties

Файл со значениями для placeholders вида ${...}. Нужен и для batch-модулей, и для SQL-консоли, если параметры подключения не захардкожены в конфиге.
Где ищется автоматически
  1. ui.defaultCredentialsFile, если он задан.
  2. gradle/credential.properties в проекте.
  3. ~/.gradle/credential.properties.
  4. Если файл не найден, его можно загрузить через UI вручную.
Минимальный пример файла
DB1_JDBC_URL=jdbc:postgresql://host:5432/db1
DB1_USERNAME=user1
DB1_PASSWORD=secret1

TARGET_JDBC_URL=jdbc:postgresql://host:5432/target_db
TARGET_USERNAME=loader
TARGET_DB_PASSWORD=target_secret
Если что-то не работает
Пустой список модулей
Проверь ui.appsRoot в ui-application.yml. Для собранного UI нужен абсолютный путь к каталогу apps.
Placeholder не найден
Проверь, что нужный ключ действительно есть в credential.properties, и что UI нашел этот файл или ты загрузил его вручную.
SQL не применяется
Проверь, не заданы ли одновременно inline SQL и SQL-файл для одного и того же уровня конфига.
Слишком долгий запрос
Проверь queryTimeoutSec в модуле или ui.sqlConsole.queryTimeoutSec в UI-конфиге.
