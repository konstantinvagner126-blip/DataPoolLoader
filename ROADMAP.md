# DataPoolLoader Roadmap

## Цель

Развивать DataPoolLoader как внутреннюю платформу для:

- подготовки пулов данных из нескольких PostgreSQL источников;
- загрузки результатов в целевую БД;
- запуска сценариев как через app-модули, так и через локальный UI;
- ручной работы с шардами через SQL-консоль;
- удобного сопровождения и расширения новыми модулями.

## Принципы развития

- `core` остается единым источником бизнес-логики;
- `apps` содержат только сценарии выгрузки и загрузки данных;
- `ui` развивается как отдельная операторская оболочка над `core`;
- новый функционал сначала делаем полезным и понятным, потом усложняем;
- все важные пользовательские сценарии должны оставаться рабочими из IDEA.

## Ближайший этап

### 1. Улучшение UI как рабочего инструмента

Цель:

- сделать UI удобным не только для запуска, но и для анализа результатов.

Задачи:

- добавить визуальный редактор конфигурации модуля;
- сделать в редакторе модуля отдельную вкладку `Форма` рядом с вкладкой `YAML`;
- показывать настройки конфига через визуальные компоненты:
  - checkbox;
  - radio button;
  - select;
  - текстовые и числовые поля;
- реализовать полную двустороннюю синхронизацию в реальном времени:
  - изменение YAML сразу обновляет форму;
  - изменение формы сразу обновляет YAML;
- в первой версии визуальной формы покрыть основные поля `app` и `target`;
- для сложных структур (`sources`, `quotas`) предусмотреть поэтапное расширение формы;
- улучшить экран истории запусков;
- добавить удобный просмотр `summary.json` в UI;
- показать summary не как raw JSON, а как структурированные блоки:
  - merged rows;
  - target status;
  - successful sources;
  - failed sources;
- улучшить отображение ошибок и предупреждений в UI;
- добавить стартовые подсказки по использованию интерфейса.

### 2. Улучшение операторского UX

Цель:

- снизить количество “технических” действий, которые пользователь должен помнить.

Задачи:

- сделать более компактную и понятную компоновку экранов;
- улучшить отображение статуса выполнения и прогресса;
- сделать более понятную навигацию между модулями, SQL-консолью и историей запусков;
- добавить короткие пояснения по `credential.properties`, timeout и output.
- сделать работу с конфигом модуля удобной для пользователей, которые не хотят редактировать YAML вручную.

## Следующий этап

### 3. Развитие SQL-консоли

Цель:

- превратить SQL-консоль в полноценный рабочий инструмент для ручной проверки и служебных операций.

Текущее состояние:

- поддерживается один SQL statement за запуск;
- данные по `SELECT` показываются отдельно по каждому source;
- остальные команды показывают статусы выполнения по source;
- есть выбор источников, timeout и отмена.

Следующие шаги:

- экспорт результатов `SELECT` из SQL-консоли;
- сохранение и повторное использование частых запросов;
- более наглядное отображение длительности и статусов выполнения;
- отдельные guardrails для опасных команд;
- затем, как отдельный большой этап, поддержка `multi-statement SQL`.

### 4. Проектирование `multi-statement SQL`

Цель:

- поддержать запуск SQL-скриптов из нескольких statement в SQL-консоли.

Пока не реализуем, но держим как следующий крупный функциональный блок.

Что нужно будет решить:

- разбор SQL-скрипта на отдельные statement;
- выполнение по каждому source по порядку;
- модель ошибок:
  - stop on first error;
  - continue on error;
- транзакционный режим:
  - один statement = одна транзакция;
  - или весь script в одной транзакции;
- отображение результата:
  - source;
  - statement index;
  - keyword;
  - status;
  - affected rows;
  - error.

## Среднесрочный этап

### 5. Надежность и эксплуатация

Цель:

- сделать поведение системы предсказуемым для команды.

Задачи:

- retention для старых output-каталогов;
- более подробный summary по длительностям:
  - export;
  - merge;
  - target import;
- единые правила работы с путями;
- дополнительные end-to-end проверки сценариев;
- улучшение startup/logging диагностики.

### 6. Конфиги и модульность

Цель:

- упростить развитие новых сценариев и их регистрацию в UI.

Задачи:

- ввести более формальный descriptor модуля;
- добавить metadata модуля:
  - id;
  - title;
  - description;
  - теги;
- улучшить шаблон генерации нового app-модуля;
- автоматически валидировать модуль при старте UI;
- уметь скрывать тестовые или служебные модули из UI.

## Долгосрочный этап

### 7. Технический долг и упрощение сопровождения

Цель:

- снизить сложность ключевых классов.

Кандидаты на разбиение:

- `core/src/main/kotlin/com/sbrf/lt/datapool/app/ApplicationRunner.kt`
- `core/src/main/kotlin/com/sbrf/lt/datapool/sqlconsole/SqlConsoleService.kt`
- `ui/src/main/kotlin/com/sbrf/lt/datapool/ui/RunManager.kt`
- `ui/src/main/kotlin/com/sbrf/lt/datapool/ui/Server.kt`

Задачи:

- выделить более узкие сервисы;
- упростить DTO и преобразования между `core` и `ui`;
- стабилизировать API между `core`, `apps` и `ui`.

### 8. Локальный запуск и сопровождение команды

Цель:

- сделать инструмент удобным не только для разработчика, который знает проект изнутри.

Задачи:

- launcher-скрипты для UI;
- более короткий onboarding в README;
- отдельный раздел “что делать после добавления нового модуля”;
- удобный сценарий первого запуска без ручного чтения кода.

## Рекомендуемый порядок реализации

### MVP+1

- визуальный редактор конфигурации модуля;
- двусторонняя синхронизация `YAML <-> форма` в реальном времени;
- история запусков в UI;
- нормальный просмотр summary;
- улучшение UX и подсказок в UI.

### MVP+2

- экспорт результатов SQL-консоли;
- guardrails для опасных SQL-команд;
- более удобное повторное использование запросов.

### MVP+3

- retention output-каталогов;
- metadata модулей;
- валидация и фильтрация модулей в UI.

### Крупный отдельный этап

- `multi-statement SQL` в SQL-консоли.

## Что считаем правильным сейчас

- не смешивать бизнес-логику с UI;
- не усложнять SQL-консоль до script-runner без отдельного проектирования;
- сначала улучшать usability и наблюдаемость;
- только после этого добавлять тяжелые функциональные расширения.
