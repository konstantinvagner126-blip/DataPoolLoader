# Architecture Rules

Дата: 2026-04-19

Назначение документа:

- зафиксировать обязательные архитектурные правила проекта;
- дать единый набор инженерных ограничений для всех следующих изменений;
- удерживать проект в локальной, устойчивой и поддерживаемой архитектуре.

Приоритет проекта:

- хорошая архитектура;
- надежность;
- расширяемость.

Эти три критерия важнее локальной скорости доставки фичи, если между ними возникает конфликт.

Этот документ нужно читать перед изменениями, которые затрагивают:

- структуру модулей;
- `core`;
- `ui-server`;
- persisted state;
- SQL-консоль;
- long-running операции;
- крупные Compose-экраны и store-слой.

Для изменений, которые затрагивают state model, дополнительно обязательна сверка с:

- [STATE_MODEL_MAP.md](/Users/kwdev/DataPoolLoader/STATE_MODEL_MAP.md)

## 1. Базовый контекст проекта

Проект рассматривается как:

- локальное инженерное приложение;
- пользователь запускает его у себя на машине;
- допускаются локальные файлы состояния и локальные runtime-допущения;
- не предполагается, что архитектура обязана решать полноценный multi-user web-сценарий.

Из этого не следует, что можно размывать архитектуру. Это означает только, что решения должны быть прагматичны и соответствовать именно локальному характеру продукта.

## 2. Базовая архитектурная оценка проекта

Этот раздел фиксирует не правила "вообще", а текущее понимание именно этого проекта.

### 2.1. Что в проекте уже хорошо

- проект реально разделен на `core`, `ui-server`, `ui-compose-shared`, `ui-compose-web`, `ui-compose-desktop`;
- доменная и execution-логика в основном остается в `core`, а не размазывается по UI;
- backend уже покрыт заметным количеством тестов;
- проект уже проходит через системный рефакторинг support-слоев и контрактов, а не только обрастает фичами.

### 2.2. Главные текущие слабости

- `ui-server` архитектурно перегружен и легко превращается в knowledge-heavy service locator;
- error model слишком грубая и не должна дальше развиваться в текущем виде;
- persisted state, runtime state, config и history еще недостаточно строго разделены;
- главный maintainability debt сейчас в giant Compose pages и толстых store-слоях;
- SQL-консоль уже является отдельной подсистемой и не должна развиваться как просто "еще один экран";
- long-running операции пока требуют постоянного safety-контроля и не считаются закрытой темой.

### 2.3. Приоритет архитектурной программы

Для этого проекта приоритет ближайшего этапа такой:

1. архитектурный порядок;
2. надежность;
3. расширяемость;
4. cleanup structural debt;
5. только после этого новые product/UX-расширения, если они не ломают первые четыре пункта.

### 2.4. Review After Giant-File Wave

По итогам этапа распила giant UI/store files зафиксировано:

- page-layer и основной store-layer приведены к состоянию shell/binding/effects/support вместо монолитных экранов;
- дальнейший распил этих же файлов нельзя продолжать по инерции;
- сам по себе размер файла больше не считается достаточным основанием для нового refactoring pass, если файл уже reviewable и держит связную ответственность.

Следующая волна изменений после этого этапа должна идти не в повторный structural split тех же экранов, а в:

- cleanup legacy;
- нормализацию state и contracts;
- reliability/safety;
- следующий верхний backlog-stream.

## 3. Главные архитектурные принципы

### 3.1. `core` владеет бизнес-логикой

В `core` должна жить:

- логика выполнения модулей;
- SQL-console execution logic;
- sync logic;
- validation;
- model transformations, не привязанные к Ktor и Compose.

В `core` не должна утекать:

- Ktor-специфика;
- Compose UI state;
- web routing;
- page navigation;
- browser-specific logic.

### 3.2. `ui-server` это boundary и orchestration layer, а не второй `core`

В `ui-server` допустимы:

- HTTP routes;
- composition root;
- persisted local state;
- orchestration long-running операций;
- runtime mode resolution;
- mapping domain <-> transport.

В `ui-server` нежелательны:

- толстая бизнес-логика;
- большие правила валидации, которые должны жить в `core`;
- разрастание route handlers;
- новые "глобальные mutable service" без жесткого объяснения причины.

### 3.3. `ui-compose-shared` это shared UI state layer, а не mini-backend

В `ui-compose-shared` допустимы:

- store;
- state;
- view models / DTO;
- UI-driven orchestration;
- shared screen contracts.

В `ui-compose-shared` нельзя бесконтрольно переносить:

- доменную бизнес-логику;
- сложную server-side политику;
- runtime ownership rules;
- data-processing логику.

### 3.4. `ui-compose-web` должен быть слоем экранов и компонентов

В `ui-compose-web` допустимы:

- page composition;
- визуальные компоненты;
- локальный interaction state;
- browser integrations;
- Monaco / DOM / navigation glue.

Недопустимо:

- превращать page-файлы в новые монолиты;
- держать в одном page-файле весь экран, бизнес-ветвление, форматирование, side effects и сервисные helper-блоки.

### 3.5. Разработка должна идти от контрактов и интерфейсов

Для заметной новой логики целевая модель такая:

- сначала определяется контракт;
- потом boundary;
- потом реализация;
- потом wiring.

Практически это означает:

- зависимости должны смотреть на интерфейсы, а не на concrete-классы, если слой имеет самостоятельную ответственность;
- orchestration-слои не должны зависеть от деталей хранения и выполнения напрямую;
- support/service/query/execution/maintenance части нужно отделять контрактами там, где это снижает связанность.

Исключения допустимы только для:

- тривиальных leaf-компонентов;
- маленьких локальных helper-объектов;
- короткоживущего кода, который не является архитектурной опорой.

Если компонент:

- участвует в lifecycle операции;
- является boundary между модулями;
- держит persisted state;
- управляет execution;
- используется несколькими слоями,

то разработка через явный контракт считается обязательной по умолчанию.

## 4. Правила про состояние

### 4.1. Любое состояние должно быть классифицировано

Текущая инженерная карта состояний проекта поддерживается в:

- [STATE_MODEL_MAP.md](/Users/kwdev/DataPoolLoader/STATE_MODEL_MAP.md)

Каждое новое состояние должно быть явно отнесено к одному из типов:

- transient runtime state;
- persisted local state;
- source of truth;
- cache;
- UI preference;
- operational history.

Если при добавлении нового состояния неясно, к какому типу оно относится, архитектурное решение считается плохим.

Если изменение добавляет новый значимый state boundary или меняет source of truth существующего состояния, соответствующее изменение в [STATE_MODEL_MAP.md](/Users/kwdev/DataPoolLoader/STATE_MODEL_MAP.md) обязательно.

### 4.2. Нельзя смешивать operational state и user preferences

Нельзя хранить в одной persisted модели одновременно:

- настройки UI;
- draft-данные пользователя;
- execution state;
- operational history;
- runtime mode decisions.

Исключения допускаются только временно и должны рассматриваться как технический долг.

### 4.3. Глобальное mutable state допускается только как осознанное локальное решение

Так как приложение локальное, полностью избежать process-local состояния нельзя.

Но любое новое глобальное mutable state должно отвечать на вопросы:

1. почему оно не может жить как локальный request/screen state;
2. почему оно не может быть persisted явно;
3. как оно восстанавливается после рестарта;
4. что будет при закрытии вкладки/экрана;
5. не смешивает ли оно UI preference и operational state.

Если ответы размытые, такое состояние добавлять нельзя.

## 5. Правила по long-running операциям

### 5.1. Long-running operations это first-class concern

Любая долгая операция должна иметь:

- явный lifecycle;
- видимый статус;
- завершение success / failure / cancel;
- понятную политику восстановления;
- понятную политику очистки.

Это относится к:

- batch run;
- DB run;
- cleanup/retention;
- SQL console execution.

### 5.2. SQL-консоль считается отдельной подсистемой

Любые изменения SQL-консоли должны оцениваться отдельно по:

- transaction safety;
- execution lifecycle;
- cancel behavior;
- pending commit / rollback behavior;
- failure scenarios.

Нельзя вносить серьезные изменения в SQL-консоль без проверки документов:

- [SQL_CONSOLE_FAILURE_SCENARIOS.md](/Users/kwdev/DataPoolLoader/SQL_CONSOLE_FAILURE_SCENARIOS.md)
- [SQL_CONSOLE_MONACO_AUTOCOMPLETE.md](/Users/kwdev/DataPoolLoader/SQL_CONSOLE_MONACO_AUTOCOMPLETE.md)

### 5.3. Безопасность БД важнее удобства UI

Если есть конфликт между:

- быстрым UX;
- и безопасным lifecycle транзакции/запуска,

приоритет всегда у безопасности.

Нельзя допускать решения, которые:

- оставляют неочевидный pending state;
- теряют контроль над transaction lifecycle;
- маскируют опасную операцию под "безобидный" UI action.

## 6. Правила по API и ошибкам

### 6.1. Route handlers должны быть тонкими

Route handler должен:

- читать вход;
- валидировать transport-level данные;
- вызывать orchestration/service;
- возвращать transport-level ответ.

Route handler не должен становиться местом, где:

- живет основная бизнес-логика;
- размазываются runtime-mode правила;
- собираются большие сценарии с несколькими ветками ответственности.

Route handler также не должен становиться местом, где concrete-реализации намертво прошиваются без контрактного слоя, если этот сценарий используется как архитектурная опора.

### 6.2. Нельзя продолжать практику `Throwable -> 400`

Новые изменения не должны усиливать эту модель.

Нужно двигаться к разделению минимум на:

- validation/business error;
- conflict/state error;
- internal error.

Даже для локального приложения грубая error model усложняет поддержку и диагностику.

## 7. Правила по размеру и структуре файлов

### 7.1. Большой файл это архитектурный сигнал

Ориентиры:

- page-файл `600+` строк: повод на декомпозицию;
- page-файл `900+` строк: декомпозиция обязательна;
- store `500+` строк: обязательная проверка на смешение ответственностей;
- support/service `300+` строк: проверка на то, не держит ли файл несколько контекстов сразу.

Это не механический линтер, но жесткий сигнал.

### 7.2. Нельзя добавлять новые монолитные страницы

Запрещено:

- добавлять новый экран на `1000+` строк;
- сливать в один файл layout, actions, sub-panels, formatting, side effects и browser glue;
- увеличивать уже крупный page-файл без попытки вынести новые блоки в отдельные компоненты.

Аналогично запрещено наращивать крупные orchestration/service/store-файлы, если проблему можно разрезать контрактами и support-слоями.

### 7.3. Нельзя уходить во второй круг распила без нового архитектурного эффекта

Если файл уже:

- обозрим при ревью;
- имеет связную ответственность;
- не смешивает несколько lifecycle-контекстов;
- не тянет orchestration обратно в page/store shell,

то его нельзя продолжать дробить только ради дальнейшего уменьшения числа строк.

Повторный refactoring pass допустим только если появился новый smell:

- снова смешались разные subflow;
- вырос knowledge-heavy orchestration;
- нарушилась контрактная граница;
- файл снова стал дорогим для локального изменения и ревью.

## 8. Правила по развитию UI

### 8.1. Экран должен состоять из feature-блоков

Для крупных экранов целевая модель такая:

- shell page;
- отдельные visual sections;
- отдельные action panels;
- маленькие reusable components;
- thin adapters к store.

### 8.2. Нельзя перегружать интерфейс лишними данными

В интерфейсе должны отображаться:

- данные, нужные для действия;
- данные, нужные для принятия решения;
- данные, нужные для контроля статуса и безопасности.

Не нужно выводить в основной экран:

- технические подсказки, если действие уже понятно из контекста;
- дублирующие статусы;
- вторичные метаданные, которые не помогают принять решение;
- визуальные блоки, которые объясняют внутреннюю механику, но не помогают работать быстрее.

Если блок:

- не влияет на действие пользователя;
- не снижает риск ошибки;
- не помогает понять текущее состояние;

то по умолчанию его не должно быть в основном интерфейсе.

### 8.3. UI не должен быть единственным местом знания о сценарии

Если важная логика существует только в page/store и не отражена на уровне `core` или `ui-server`, это плохая архитектура.

Особенно это критично для:

- SQL console execution;
- module run lifecycle;
- cleanup/retention;
- config mutation rules.

## 9. Правила по тестам

### 9.1. Backend-изменения без тестов допустимы только для тривиальных правок

Если меняется:

- lifecycle операции;
- persisted state;
- SQL execution behavior;
- cleanup/retention logic;
- route behavior;

нужен тест или очень сильное объяснение, почему тест не добавлен.

### 9.2. Для frontend крупный UX-change должен иметь хотя бы косвенную страховку

Если невозможно быстро добавить UI-тест, должно быть хотя бы одно из:

- store-level verification;
- server contract test;
- compile-time safety плюс ручной сценарий;
- отдельный failure-scenario doc update.

## 10. Правила по качеству кода

### 10.1. На код нужно смотреть sonar-like взглядом

При любом заметном изменении нужно проверять не только “работает / не работает”, но и качество самого кода.

Обязательные вопросы:

- не появился ли дублирующийся код;
- не выросла ли скрытая связанность;
- не стал ли метод, класс, store или page чрезмерно большим;
- не появилась ли лишняя сложность в ветвлениях и lifecycle;
- не остался ли мертвый код, старый адаптер или legacy-хвост;
- не маскируется ли архитектурная проблема под "маленькую локальную правку".

### 10.2. Признаки плохого качества считаются дефектами, а не косметикой

К таким признакам относятся:

- дублирование логики;
- мертвый код;
- неиспользуемые модели, поля, CSS-классы и helper-ы;
- giant methods / giant files;
- бессистемное разрастание store/service/page;
- неочевидные названия и скрытые side effects;
- подавление ошибок или слишком широкие `catch` без причины.

Если такие проблемы видны, их нельзя игнорировать только потому, что "фича и так работает".

### 10.3. Любая правка должна либо улучшать качество, либо как минимум не ухудшать его

Нельзя:

- добавлять новый долг в уже перегруженный участок без явной фиксации этого долга;
- усиливать дублирование;
- оставлять после себя неиспользуемые куски кода;
- наращивать размер файлов и классов без попытки декомпозиции.

## 11. Практические запреты

Явно запрещено:

- добавлять новые глобальные mutable singleton-like сервисы без justification;
- складывать новую orchestration-логику в routes;
- смешивать UI preferences и operational state;
- расширять SQL-консоль без проверки safety;
- плодить новые giant Compose pages;
- делать скрытые опасные действия в БД без явной визуальной модели состояния.

## 12. Обязательный check-list перед архитектурно заметным изменением

Перед заметным изменением нужно ответить:

1. В каком модуле должна жить логика на самом деле?
2. Не тяну ли я domain-логику в `ui-server` или UI?
3. Как классифицируется новое состояние?
4. Что произойдет после рестарта?
5. Что произойдет при закрытии вкладки или экрана?
6. Не увеличиваю ли я giant-file debt?
7. Не ухудшаю ли я качество кода по sonar-like критериям?
8. Нужен ли новый test или update safety-документации?

Если хотя бы на два пункта ответ неясен, изменение нужно переосмыслить до реализации.

## 13. Правило фиксации изменений

Работа по backlog должна фиксироваться регулярно, а не откладываться до большого коммита в конце длинной серии задач.

Обязательное правило:

- после реализации каждой третьей задачи из рабочего [BACKLOG.md](/Users/kwdev/DataPoolLoader/BACKLOG.md) изменения нужно:
  - закоммитить;
  - запушить в рабочую ветку.

Цель правила:

- уменьшить риск большого неуправляемого change set;
- упростить ревью и откат;
- не держать архитектурно заметные изменения слишком долго только локально.

## 14. Правило работы только через backlog

Нельзя выполнять новые доработки вне рабочего [BACKLOG.md](/Users/kwdev/DataPoolLoader/BACKLOG.md).

Если появляется новая идея, запрос или доработка, правильный порядок всегда такой:

1. сформулировать задачу;
2. добавить ее в backlog;
3. согласовать приоритет и ожидаемый результат;
4. только после этого переходить к реализации в коде.

Это правило нужно для того, чтобы:

- не размывать архитектурную программу;
- не плодить хаотичные изменения;
- не смешивать срочные идеи с уже принятым планом развития;
- сохранять управляемость change set и истории решений.

Исключения не предполагаются. Даже если доработка кажется маленькой, она все равно сначала фиксируется в backlog.

## 15. Правило обязательного архитектурного review после крупных этапов

После завершения каждого большого этапа backlog нужно выполнять отдельный архитектурный review текущего состояния проекта.

Под большим этапом понимается:

- завершение заметной волны рефакторинга;
- закрытие крупного блока `P0` или `P1`;
- серия изменений, которая существенно меняет границы, state model, execution lifecycle или UI-структуру.

Цель review:

- проверить, не разошлись ли код и архитектурные правила;
- обновить архитектурную оценку проекта;
- выявить новый structural debt;
- при необходимости скорректировать backlog и сами правила.

Архитектурный review после большого этапа считается обязательной частью процесса, а не дополнительной активностью "если останется время".

У проекта хороший фундамент, но он уже требует не только разработки функций, а постоянного архитектурного контроля.

Цель этих правил:

- не тормозить разработку;
- а удерживать проект в состоянии, где он остается локальным, надежным и поддерживаемым инженерным приложением.
