# Architecture Rules

Дата: 2026-04-19

Назначение документа:

- зафиксировать обязательные архитектурные правила проекта;
- дать единый набор инженерных ограничений для всех следующих изменений;
- удерживать проект в локальной, устойчивой и поддерживаемой архитектуре.

Приоритет проекта:

- хорошая архитектура;
- надежность;
- расширяемость.

Эти три критерия важнее локальной скорости доставки фичи, если между ними возникает конфликт.

Этот документ нужно читать перед изменениями, которые затрагивают:

- структуру модулей;
- `core`;
- `ui-server`;
- persisted state;
- SQL-консоль;
- long-running операции;
- крупные Compose-экраны и store-слой.

## 1. Базовый контекст проекта

Проект рассматривается как:

- локальное инженерное приложение;
- пользователь запускает его у себя на машине;
- допускаются локальные файлы состояния и локальные runtime-допущения;
- не предполагается, что архитектура обязана решать полноценный multi-user web-сценарий.

Из этого не следует, что можно размывать архитектуру. Это означает только, что решения должны быть прагматичны и соответствовать именно локальному характеру продукта.

## 2. Главные архитектурные принципы

### 2.1. `core` владеет бизнес-логикой

В `core` должна жить:

- логика выполнения модулей;
- SQL-console execution logic;
- sync logic;
- validation;
- model transformations, не привязанные к Ktor и Compose.

В `core` не должна утекать:

- Ktor-специфика;
- Compose UI state;
- web routing;
- page navigation;
- browser-specific logic.

### 2.2. `ui-server` это boundary и orchestration layer, а не второй `core`

В `ui-server` допустимы:

- HTTP routes;
- composition root;
- persisted local state;
- orchestration long-running операций;
- runtime mode resolution;
- mapping domain <-> transport.

В `ui-server` нежелательны:

- толстая бизнес-логика;
- большие правила валидации, которые должны жить в `core`;
- разрастание route handlers;
- новые "глобальные mutable service" без жесткого объяснения причины.

### 2.3. `ui-compose-shared` это shared UI state layer, а не mini-backend

В `ui-compose-shared` допустимы:

- store;
- state;
- view models / DTO;
- UI-driven orchestration;
- shared screen contracts.

В `ui-compose-shared` нельзя бесконтрольно переносить:

- доменную бизнес-логику;
- сложную server-side политику;
- runtime ownership rules;
- data-processing логику.

### 2.4. `ui-compose-web` должен быть слоем экранов и компонентов

В `ui-compose-web` допустимы:

- page composition;
- визуальные компоненты;
- локальный interaction state;
- browser integrations;
- Monaco / DOM / navigation glue.

Недопустимо:

- превращать page-файлы в новые монолиты;
- держать в одном page-файле весь экран, бизнес-ветвление, форматирование, side effects и сервисные helper-блоки.

### 2.5. Разработка должна идти от контрактов и интерфейсов

Для заметной новой логики целевая модель такая:

- сначала определяется контракт;
- потом boundary;
- потом реализация;
- потом wiring.

Практически это означает:

- зависимости должны смотреть на интерфейсы, а не на concrete-классы, если слой имеет самостоятельную ответственность;
- orchestration-слои не должны зависеть от деталей хранения и выполнения напрямую;
- support/service/query/execution/maintenance части нужно отделять контрактами там, где это снижает связанность.

Исключения допустимы только для:

- тривиальных leaf-компонентов;
- маленьких локальных helper-объектов;
- короткоживущего кода, который не является архитектурной опорой.

Если компонент:

- участвует в lifecycle операции;
- является boundary между модулями;
- держит persisted state;
- управляет execution;
- используется несколькими слоями,

то разработка через явный контракт считается обязательной по умолчанию.

## 3. Правила про состояние

### 3.1. Любое состояние должно быть классифицировано

Каждое новое состояние должно быть явно отнесено к одному из типов:

- transient runtime state;
- persisted local state;
- source of truth;
- cache;
- UI preference;
- operational history.

Если при добавлении нового состояния неясно, к какому типу оно относится, архитектурное решение считается плохим.

### 3.2. Нельзя смешивать operational state и user preferences

Нельзя хранить в одной persisted модели одновременно:

- настройки UI;
- draft-данные пользователя;
- execution state;
- operational history;
- runtime mode decisions.

Исключения допускаются только временно и должны рассматриваться как технический долг.

### 3.3. Глобальное mutable state допускается только как осознанное локальное решение

Так как приложение локальное, полностью избежать process-local состояния нельзя.

Но любое новое глобальное mutable state должно отвечать на вопросы:

1. почему оно не может жить как локальный request/screen state;
2. почему оно не может быть persisted явно;
3. как оно восстанавливается после рестарта;
4. что будет при закрытии вкладки/экрана;
5. не смешивает ли оно UI preference и operational state.

Если ответы размытые, такое состояние добавлять нельзя.

## 4. Правила по long-running операциям

### 4.1. Long-running operations это first-class concern

Любая долгая операция должна иметь:

- явный lifecycle;
- видимый статус;
- завершение success / failure / cancel;
- понятную политику восстановления;
- понятную политику очистки.

Это относится к:

- batch run;
- DB run;
- cleanup/retention;
- SQL console execution.

### 4.2. SQL-консоль считается отдельной подсистемой

Любые изменения SQL-консоли должны оцениваться отдельно по:

- transaction safety;
- execution lifecycle;
- cancel behavior;
- pending commit / rollback behavior;
- failure scenarios.

Нельзя вносить серьезные изменения в SQL-консоль без проверки документов:

- [SQL_CONSOLE_FAILURE_SCENARIOS.md](/Users/kwdev/DataPoolLoader/SQL_CONSOLE_FAILURE_SCENARIOS.md)
- [SQL_CONSOLE_MONACO_AUTOCOMPLETE.md](/Users/kwdev/DataPoolLoader/SQL_CONSOLE_MONACO_AUTOCOMPLETE.md)

### 4.3. Безопасность БД важнее удобства UI

Если есть конфликт между:

- быстрым UX;
- и безопасным lifecycle транзакции/запуска,

приоритет всегда у безопасности.

Нельзя допускать решения, которые:

- оставляют неочевидный pending state;
- теряют контроль над transaction lifecycle;
- маскируют опасную операцию под "безобидный" UI action.

## 5. Правила по API и ошибкам

### 5.1. Route handlers должны быть тонкими

Route handler должен:

- читать вход;
- валидировать transport-level данные;
- вызывать orchestration/service;
- возвращать transport-level ответ.

Route handler не должен становиться местом, где:

- живет основная бизнес-логика;
- размазываются runtime-mode правила;
- собираются большие сценарии с несколькими ветками ответственности.

Route handler также не должен становиться местом, где concrete-реализации намертво прошиваются без контрактного слоя, если этот сценарий используется как архитектурная опора.

### 5.2. Нельзя продолжать практику `Throwable -> 400`

Новые изменения не должны усиливать эту модель.

Нужно двигаться к разделению минимум на:

- validation/business error;
- conflict/state error;
- internal error.

Даже для локального приложения грубая error model усложняет поддержку и диагностику.

## 6. Правила по размеру и структуре файлов

### 6.1. Большой файл это архитектурный сигнал

Ориентиры:

- page-файл `600+` строк: повод на декомпозицию;
- page-файл `900+` строк: декомпозиция обязательна;
- store `500+` строк: обязательная проверка на смешение ответственностей;
- support/service `300+` строк: проверка на то, не держит ли файл несколько контекстов сразу.

Это не механический линтер, но жесткий сигнал.

### 6.2. Нельзя добавлять новые монолитные страницы

Запрещено:

- добавлять новый экран на `1000+` строк;
- сливать в один файл layout, actions, sub-panels, formatting, side effects и browser glue;
- увеличивать уже крупный page-файл без попытки вынести новые блоки в отдельные компоненты.

Аналогично запрещено наращивать крупные orchestration/service/store-файлы, если проблему можно разрезать контрактами и support-слоями.

## 7. Правила по развитию UI

### 7.1. Экран должен состоять из feature-блоков

Для крупных экранов целевая модель такая:

- shell page;
- отдельные visual sections;
- отдельные action panels;
- маленькие reusable components;
- thin adapters к store.

### 7.2. Нельзя перегружать интерфейс лишними данными

В интерфейсе должны отображаться:

- данные, нужные для действия;
- данные, нужные для принятия решения;
- данные, нужные для контроля статуса и безопасности.

Не нужно выводить в основной экран:

- технические подсказки, если действие уже понятно из контекста;
- дублирующие статусы;
- вторичные метаданные, которые не помогают принять решение;
- визуальные блоки, которые объясняют внутреннюю механику, но не помогают работать быстрее.

Если блок:

- не влияет на действие пользователя;
- не снижает риск ошибки;
- не помогает понять текущее состояние;

то по умолчанию его не должно быть в основном интерфейсе.

### 7.3. UI не должен быть единственным местом знания о сценарии

Если важная логика существует только в page/store и не отражена на уровне `core` или `ui-server`, это плохая архитектура.

Особенно это критично для:

- SQL console execution;
- module run lifecycle;
- cleanup/retention;
- config mutation rules.

## 8. Правила по тестам

### 8.1. Backend-изменения без тестов допустимы только для тривиальных правок

Если меняется:

- lifecycle операции;
- persisted state;
- SQL execution behavior;
- cleanup/retention logic;
- route behavior;

нужен тест или очень сильное объяснение, почему тест не добавлен.

### 8.2. Для frontend крупный UX-change должен иметь хотя бы косвенную страховку

Если невозможно быстро добавить UI-тест, должно быть хотя бы одно из:

- store-level verification;
- server contract test;
- compile-time safety плюс ручной сценарий;
- отдельный failure-scenario doc update.

## 9. Практические запреты

Явно запрещено:

- добавлять новые глобальные mutable singleton-like сервисы без justification;
- складывать новую orchestration-логику в routes;
- смешивать UI preferences и operational state;
- расширять SQL-консоль без проверки safety;
- плодить новые giant Compose pages;
- делать скрытые опасные действия в БД без явной визуальной модели состояния.

## 10. Обязательный check-list перед архитектурно заметным изменением

Перед заметным изменением нужно ответить:

1. В каком модуле должна жить логика на самом деле?
2. Не тяну ли я domain-логику в `ui-server` или UI?
3. Как классифицируется новое состояние?
4. Что произойдет после рестарта?
5. Что произойдет при закрытии вкладки или экрана?
6. Не увеличиваю ли я giant-file debt?
7. Нужен ли новый test или update safety-документации?

Если хотя бы на два пункта ответ неясен, изменение нужно переосмыслить до реализации.

## 11. Итог

У проекта хороший фундамент, но он уже требует не только разработки функций, а постоянного архитектурного контроля.

Цель этих правил:

- не тормозить разработку;
- а удерживать проект в состоянии, где он остается локальным, надежным и поддерживаемым инженерным приложением.
